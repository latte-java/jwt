# JWT 7.0 Clean-Break Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan checkpoint-by-checkpoint. Within each checkpoint, follow superpowers:test-driven-development (red → green → refactor).

**Goal:** Full rewrite of the FusionAuth/Latte JWT library to the 7.0 architecture defined in `specs/7.0-architecture.md` — zero required dependencies, immutable models, pluggable JSON, hardened decoder, no `sun.*` private APIs.

**Architecture:** See `specs/7.0-architecture.md` (the spec is the contract; this plan only sequences the work). Foundational interfaces first (Algorithm/KeyType, JSONProcessor), then models (JWT/Header), then crypto (Signer/Verifier with the CVE-adjacent surface in §6), then glue (Encoder/Decoder/VerifierResolver), then JWK/JWKS, then PEM/DER/X.509. Tests are TDD per the §14 use-case lists.

**Tech Stack:** Java 17+, TestNG with `@DataProvider`, Maven. New files use the MIT license header. Jackson removed from compile scope (test-only if retained for `JSONProcessor` integration tests).

---

## Plan Conventions

- **The spec is the contract.** This plan does not repeat code, contracts, or test enumerations from `specs/7.0-architecture.md`. Each task references the relevant `§` and a concrete file list. Engineers MUST read the referenced spec section before writing code.
- **TDD per task.** Each task lists the failing test(s) to write first, then the implementation, then the run command, then commit. Test names map 1:1 to the `// Use case:` bullets in spec §14.
- **DataProviders.** When the spec §14 bullet says "DataProvider over {…}" or describes dimensional variants, write a single `@Test(dataProvider = "…")` with a `@DataProvider` method. Default to DataProvider whenever a test varies a single parameter (algorithm family, malformed-input variant, boundary value).
- **Commit cadence.** One commit per task — message format: `feat(7.0): <task summary>` (or `test(7.0)`, `refactor(7.0)`, `chore(7.0)`). All commits in this branch.
- **Settled questions.** Spec §16 closes ~30 design questions. Do not reopen them. If a real spec gap is discovered during implementation, update `specs/7.0-architecture.md` and flag the change in the checkpoint summary rather than deviating silently.
- **License header.** All new files start with the MIT license block (see `specs/7.0-architecture.md` line 8). Modified existing files keep their existing Apache-2.0 header.
- **Verification before completion.** Each checkpoint ends with `mvn test` (or a scoped variant for very large checkpoints) producing a passing run, with output shown in the checkpoint summary. No "should pass" / "tests look green" assertions without `mvn` output.

## Checkpoint 2 — JSONProcessor + LatteJSONProcessor + CanonicalJSONWriter

**Spec:** §4 (interface, defaults, internal canonical writer reference), §5 (decoder defenses table — `maxNestingDepth`, `maxNumberLength`, `allowDuplicateJSONKeys`), §10 (CanonicalJSONWriter contract).

**Files:**

- Create: `src/main/java/org/lattejava/jwt/LatteJSONProcessor.java` — built-in implementation. Constructor takes `(int maxNestingDepth, int maxNumberLength, boolean allowDuplicateJSONKeys)`; default constructor uses spec defaults.
- Create: `src/main/java/org/lattejava/jwt/internal/CanonicalJSONWriter.java` — package-private. ~40 LOC. Lex-sorted keys, no whitespace, supports only String/Number/Boolean/null per RFC 7638 §3.2.
- Create: `src/test/java/org/lattejava/jwt/LatteJSONProcessorTest.java` — covers every §14 "LatteJSONProcessor" use case as DataProvider-driven tests where the parameter varies (malformed inputs, JSON type round-trip, depth boundary, number-length boundary).
- Create: `src/test/java/org/lattejava/jwt/internal/CanonicalJSONWriterTest.java` — verifies lex ordering, whitespace absence, RFC 7638 §3.1 example output, RFC 8037 §A.3 example output.

**Tasks:**

1. **Red:** Write `LatteJSONProcessorTest` — single-element round-trip per JSON type (DataProvider over JSON-type), nested structure round-trip, Unicode-escape handling (DataProvider over edge cases including surrogate pairs and control characters), empty object/array, BigInteger > Long.MAX_VALUE, BigDecimal high precision, malformed input variants (DataProvider over: unterminated string, trailing comma, invalid escape, truncated input), duplicate keys default-rejected, duplicate keys accepted when `allowDuplicateJSONKeys=true`, depth = 16 accepted / depth = 17 rejected (DataProvider on the boundary), digit-run = 1000 accepted / 1001 rejected (DataProvider on number-length boundary covering integer, decimal, integer-with-exponent in both header and payload positions).

2. Run `mvn test -Dtest=LatteJSONProcessorTest`. Expected: COMPILE FAILURE (no class).

3. **Green:** Implement `LatteJSONProcessor`. Implementation notes:
   - Single-pass recursive-descent parser. Track depth and reject when `> maxNestingDepth`.
   - When parsing a number, count the digit-run length (integer + decimal + exponent digits, sign chars excluded) before invoking `new BigInteger(...)` or `new BigDecimal(...)`. Throw `JSONProcessingException` if `> maxNumberLength`.
   - When parsing an object, use `LinkedHashMap`. Reject duplicate keys by checking `containsKey` before put when `allowDuplicateJSONKeys=false`.
   - Top-level non-object → `JSONProcessingException` (per spec §4 javadoc on `deserialize`).
   - Writer: handle `String`, `Number` (Integer/Long/BigInteger/Double/BigDecimal), `Boolean`, `null`, `Map`, `List`. JSON string escaping per RFC 8259 §7 (escape `"`, `\`, `\b`, `\f`, `\n`, `\r`, `\t`, control chars `\u0000`-`\u001F`).
   - Always UTF-8 (`StandardCharsets.UTF_8`).

4. Run `mvn test -Dtest=LatteJSONProcessorTest`. Expected: PASS.

5. **Red:** Write `CanonicalJSONWriterTest`:
   - RFC 7638 §3.1 RSA example: input `{"e":"AQAB","kty":"RSA","n":"0vx7..."}` (insertion order need not be lex order in the input map) produces the lex-ordered RFC 7638 output bytes verbatim.
   - RFC 8037 §A.3 OKP example: same, for the Ed25519 thumbprint test vector.
   - Whitespace absent.
   - Lex ordering: keys are sorted by Unicode code-point ordering (test with non-ASCII keys to confirm code-point not collation).
   - Unsupported value type (e.g., `Map` value) throws `IllegalArgumentException` — the writer is intentionally narrow.

6. Run `mvn test -Dtest=CanonicalJSONWriterTest`. Expected: COMPILE FAILURE.

7. **Green:** Implement `CanonicalJSONWriter`. Static method `byte[] write(Map<String, Object> input)` is sufficient (no instance state).

8. Run `mvn test -Dtest=CanonicalJSONWriterTest`. Expected: PASS.

9. Commit: `feat(7.0): add JSONProcessor SPI, LatteJSONProcessor, CanonicalJSONWriter`.

## Checkpoint 4 — Signer / Verifier Crypto Core

**Spec:** §6 in full — Signer/Verifier interfaces, key-length enforcement table, HMAC constant-time contract, ECDSA DER↔JOSE conversion contract, RSASSA-PSS PSSParameterSpec contract, EC public-key on-curve revalidation, thread-safety contract, `kid` binding.

This checkpoint is the CVE-adjacent surface. Each crypto family gets its own subset of tasks; tests are TDD-driven and DataProvider-organized per spec §14.

**Files:**
- Replace: `src/main/java/org/lattejava/jwt/Signer.java` — interface per spec §6 (`algorithm()`, `sign(byte[])`, `default kid()`).
- Replace: `src/main/java/org/lattejava/jwt/Verifier.java` — interface per spec §6 (`canVerify`, `verify(Algorithm, byte[], byte[])`).
- Replace: all classes in `src/main/java/org/lattejava/jwt/algorithm/{hmac,rsa,ec,ed}/`. Each Signer/Verifier:
  - Maps JWA `Algorithm` → JCA string internally.
  - Constructs a fresh `Mac` / `Signature` per `sign`/`verify` call (spec §6 "Signer / Verifier Thread-Safety").
  - Enforces minimum key length at construction (`InvalidKeyLengthException`).
  - HMAC uses `MessageDigest.isEqual` for the comparison.
  - EC adds DER↔JOSE conversion + caller-key revalidation via `KeyFactory` round-trip.
  - PSS sets explicit `PSSParameterSpec`.
- Create: `src/main/java/org/lattejava/jwt/internal/JOSEConverter.java` — package-private. Methods:
  - `static byte[] derToJose(byte[] der, int curveIntLength)` — strips DER, left-pads, concatenates.
  - `static byte[] joseToDer(byte[] jose, int curveIntLength)` — splits, re-encodes as DER `SEQUENCE { INTEGER, INTEGER }` with leading-zero handling.
  - Both throw `JWTSigningException` / `InvalidJWTException` per spec §6.
- Create: `src/test/java/org/lattejava/jwt/internal/JOSEConverterTest.java` — round-trip per curve length (32/48/66), high-bit-set integers, leading-zero integers, malformed-DER rejection, wrong-length JOSE rejection.
- Replace: existing tests in `src/test/java/org/lattejava/jwt/algorithm/` to drive the new contracts. Existing tests are largely usable; updates focus on:
  - Switching `Signer.sign(String)` to `Signer.sign(byte[])` call sites.
  - Adding the new contract tests (DER↔JOSE round-trip, PSS forged-salt rejection, EC off-curve rejection, key-length rejection, thread-safety).
- Create: `src/test/java/org/lattejava/jwt/algorithm/SignerVerifierThreadSafetyTest.java` — DataProvider over every built-in `Signer` family. 32-thread concurrent sign + verify; assert no exceptions and all signatures verify.
- **Preserve from old `JWTTest`:** Four ECDSA r/s extraction fuzz tests deleted in Checkpoint 3 must be re-introduced at the Signer/Verifier level. Add to `ECSignerTest` (or a new `ECDSASignatureFuzzTest.java`):
  - `test_ec_p256_signature_fuzz` — `@Test(invocationCount = 2_000)` — generate fresh P-256 keypair, sign+verify random 32-byte payload, assert verify succeeds. Catches r/s leading-zero-padding bugs in DER↔JOSE that only show up probabilistically.
  - `test_ec_p384_signature_fuzz` — same for P-384, 2000 invocations.
  - `test_ec_p521_signature_fuzz` — same for P-521, 2000 invocations.
  - `test_ec_p521_external_key_signature_fuzz` — same for P-521 using an OpenSSL-generated key from the test resources (carrying forward the original `test_openssl_keys_p_521` coverage), 2000 invocations.
  - These exist specifically because high-bit-set r/s values arise stochastically; reducing the invocation count reduces detection probability. Do not lower below 2000.

**Tasks (per crypto family — repeat the TDD cycle for each):**

For **each** of: HMAC, RSA (PKCS#1 v1.5), RSA-PSS, EC (ES256/384/512/256K), EdDSA (Ed25519/Ed448):

1. **Red:** Write the family's test class (re-using existing where possible, extended per §14):
   - Sign+verify round-trip per algorithm (DataProvider over algorithm constants in the family).
   - Wrong-key rejection via `InvalidKeyLengthException` (DataProvider over below-min sizes).
   - Cross-curve rejection (`ES256` with P-384) via `InvalidKeyTypeException`.
   - Family-specific contract: ECDSA round-trip through DER↔JOSE (DataProvider over high-bit-set, leading-zero); PSS rejection of non-RFC salt-length signature; HMAC constant-time (assert `MessageDigest.isEqual` is the comparison method — verify by source inspection or by an assertion that timing variance is below a threshold under known-wrong signatures across N iterations).
