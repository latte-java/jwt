| Field | Value |

## Problem statement

The library presents three concepts to users — **JWT**, **JWKS**, and **OpenID Connect Discovery** — but only two of them are first-class types today. OIDC discovery is implicit: it happens inside `JSONWebKeySetHelper` when the caller asks for keys by issuer or by well-known URL, and the helper extracts `jwks_uri` and discards everything else. `JWKSource` re-does the same hop on every refresh.

There is an `AuthorizationServerMetaData` POJO sitting in `org.lattejava.jwt.oauth2` that models RFC 8414 (a strict OAuth 2.0 subset of OIDC discovery), with a sibling `ServerMetaDataHelper` to fetch it. Neither is wired into `JWKSource`, neither covers the OIDC superset, and the OAuth-only metadata they expose is out of scope for a JWT library.

The result: a caller who wants the OIDC configuration *and* JWKS fetching has to either (a) hand-walk discovery to extract `jwks_uri` themselves, then call `JWKSource.fromJWKS(...)`, or (b) call `JWKSource.fromIssuer(...)` and accept that the discovery doc is fetched, parsed, and thrown away on every refresh.

Naming compounds the friction: `JWKSource` means "self-refreshing remote-backed JWK set with a `VerifierResolver` interface" — but the spec word for "JSON Web Key Set" is `JWKS`. Calling the class `JWKS` matches both the spec and the user's mental model, and lets the same object serve as a simple `kid → JSONWebKey` lookup in addition to its resolver behavior.

## Goals

1. Promote OIDC discovery to a first-class type: `OpenIDConnectConfiguration`, fetched via `OpenIDConnect.discover(...)` / `discoverFromWellKnown(...)`. Useful as a standalone object outside JWT verification.
2. Rename `JWKSource` to `JWKS`; add raw-JWK lookup (`get(String)`, `keys()`, `keyIds()`) and a `fromConfiguration(OpenIDConnectConfiguration)` factory; add a static, non-refreshing `JWKS.of(...)` factory.
3. Collapse the standalone JWKS / discovery helpers into the new types. One class to look at when finding a JWKS or a discovery operation.
4. Replace process-global hardening config with per-instance / per-call `FetchLimits`. No more global mutable state.
5. Carry forward today's hardening defaults exactly (limits, response cap, redirect cap, JSON parse caps).

## Non-goals

- No changes to `JWT`, `Header`, `JWTDecoder`, `JWTEncoder`, the `Signer` / `Verifier` family, the algorithm packages, or the zero-dep stance.
- No automatic re-discovery if `jwks_uri` changes server-side. Discovery is one-shot at construction time. Callers who know the server moved must rebuild.
- No support for the RFC 8414 OAuth-only metadata endpoint (`/.well-known/oauth-authorization-server`) as a first-class operation. Callers with pure-OAuth servers fetch the JSON themselves and feed `jwks_uri` into `JWKS.fromJWKS(...)`.

## 1. `OpenIDConnectConfiguration`

A new top-level immutable POJO at `org.lattejava.jwt.OpenIDConnectConfiguration`.

### 1.1 Shape

Models the OpenID Connect Discovery 1.0 Provider Metadata superset, plus the RFC 8414 Authorization Server Metadata fields, plus the routinely-deployed OIDC extension fields (RP-Initiated Logout, Session Management, Front-/Back-Channel Logout). Every field is nullable — a pure-OAuth response will have null OIDC-specific fields, and an OIDC response without RFC 8414 extras will have null introspection / revocation fields.

Typed accessors:

| Source | Fields |
|---|---|
| OIDC Core (REQUIRED) | `issuer`, `authorizationEndpoint`, `tokenEndpoint`, `jwksURI`, `responseTypesSupported`, `subjectTypesSupported`, `idTokenSigningAlgValuesSupported` |
| OIDC Core (RECOMMENDED) | `userinfoEndpoint`, `registrationEndpoint`, `scopesSupported`, `claimsSupported` |
| OIDC Core (OPTIONAL) | `responseModesSupported`, `grantTypesSupported`, `acrValuesSupported`, `idTokenEncryptionAlgValuesSupported`, `idTokenEncryptionEncValuesSupported`, `userinfoSigningAlgValuesSupported`, `userinfoEncryptionAlgValuesSupported`, `userinfoEncryptionEncValuesSupported`, `requestObjectSigningAlgValuesSupported`, `requestObjectEncryptionAlgValuesSupported`, `requestObjectEncryptionEncValuesSupported`, `tokenEndpointAuthMethodsSupported`, `tokenEndpointAuthSigningAlgValuesSupported`, `displayValuesSupported`, `claimTypesSupported`, `serviceDocumentation`, `claimsLocalesSupported`, `uiLocalesSupported`, `claimsParameterSupported`, `requestParameterSupported`, `requestURIParameterSupported`, `requireRequestURIRegistration`, `opPolicyURI`, `opTosURI` |
| OIDC extensions | `endSessionEndpoint` (RP-Initiated Logout 1.0), `checkSessionIframe` (Session Management 1.0), `frontchannelLogoutSupported`, `frontchannelLogoutSessionSupported`, `backchannelLogoutSupported`, `backchannelLogoutSessionSupported` |
| RFC 8414 additions | `introspectionEndpoint`, `introspectionEndpointAuthMethodsSupported`, `introspectionEndpointAuthSigningAlgValuesSupported`, `revocationEndpoint`, `revocationEndpointAuthMethodsSupported`, `revocationEndpointAuthSigningAlgValuesSupported`, `codeChallengeMethodsSupported` |

Anything else (proprietary / vendor-specific / draft-spec / unknown) lands in `otherClaims() : Map<String, Object>` and is preserved through round-trip serialization.

Acronym casing follows the project rule: `jwksURI()`, `opPolicyURI()`, `opTosURI()`, `requestURIParameterSupported()`, `requireRequestURIRegistration()`. (The existing `AuthorizationServerMetaData` had `jwksUri()` / `opPolicyUri()` / `opTosUri()`, which violated the rule; the new POJO fixes this.)

### 1.2 Construction and serialization

```java
public final class OpenIDConnectConfiguration {
  // Builder is the only public path to construction; constructor is private.
  public static Builder builder();

  // Parse a discovery JSON document already in Map form.
  public static OpenIDConnectConfiguration fromMap(Map<String, Object> map);

  // Round-trip back to a JSON-serializable Map.
  public Map<String, Object> toSerializableMap();

  public String toJSON();          // serializes via LatteJSONProcessor
  @Override public String toString();   // == toJSON()
  @Override public boolean equals(Object o);
  @Override public int hashCode();
}
```

The builder is reusable: each `build()` returns a fresh immutable instance with independent collection copies. `fromMap` rejects non-string elements in array-of-string fields with `IllegalArgumentException` (matching today's `AuthorizationServerMetaData.fromMap` behavior).

### 1.3 Exception type

A new `org.lattejava.jwt.OpenIDConnectException extends RuntimeException` covers all discovery-fetch failures (network, non-2xx, JSON parse, missing `jwks_uri`, missing `issuer`, oversize response, redirect overflow, etc.). It does **not** extend `JWTException` — discovery is not a JWT operation, and putting it under `JWTException` would mislead `catch` blocks that target JWT-specific failures.

## 2. `OpenIDConnect.discover` entry points

Added to the existing `org.lattejava.jwt.OpenIDConnect` class, alongside the existing `at_hash` / `c_hash` static helpers (no rename, no removal).

```java
public static OpenIDConnectConfiguration discover(String issuer);
public static OpenIDConnectConfiguration discover(String issuer, Consumer<HttpURLConnection> customizer);
public static OpenIDConnectConfiguration discover(String issuer, FetchLimits limits, Consumer<HttpURLConnection> customizer);

public static OpenIDConnectConfiguration discoverFromWellKnown(String wellKnownURL);
public static OpenIDConnectConfiguration discoverFromWellKnown(String wellKnownURL, Consumer<HttpURLConnection> customizer);
public static OpenIDConnectConfiguration discoverFromWellKnown(String wellKnownURL, FetchLimits limits, Consumer<HttpURLConnection> customizer);
```

`discover(issuer)` trims a trailing slash on the issuer and appends `/.well-known/openid-configuration` (matches today's `JSONWebKeySetHelper.retrieveKeysFromIssuer` behavior). `discoverFromWellKnown(url)` takes a fully-qualified URL — the same overload is the right entry point for an RFC 8414 server's `/.well-known/oauth-authorization-server` URL, since the JSON shape is a subset and every OIDC-specific accessor on `OpenIDConnectConfiguration` simply returns null.

The `Consumer<HttpURLConnection>` is the same customizer hook today's helpers expose (auth headers, custom timeouts, proxies). Omitting it leaves the connection at its defaults.

When `FetchLimits` is omitted, `FetchLimits.defaults()` applies. See §4.

A discovery call performs exactly one HTTP request. The response is parsed once, fields land in the typed accessors of `OpenIDConnectConfiguration`, and the object is returned. No caching is layered into this method — the caller decides how long to hold the configuration. (`JWKS.fromConfiguration(cfg)` accepts the same configuration directly.)

## 3. `JWKS` (replaces `JWKSource`)

Renamed type at `org.lattejava.jwt.jwks.JWKS`. Implements `VerifierResolver` and `AutoCloseable`, same as today's `JWKSource`.

### 3.1 Static factories

```java
public final class JWKS implements VerifierResolver, AutoCloseable {
  // Remote-backed, lifecycle-managed
  public static Builder fromIssuer(String issuer);
  public static Builder fromWellKnown(String wellKnownURL);                 // renamed from fromWellKnownConfiguration
  public static Builder fromJWKS(String jwksURL);
  public static Builder fromConfiguration(OpenIDConnectConfiguration cfg);  // NEW

  // Static, non-refreshing — no Builder
  public static JWKS of(List<JSONWebKey> keys);                             // NEW
  public static JWKS of(JSONWebKey... keys);                                // NEW

  // One-shot fetch — no instance, no lifecycle
  public static List<JSONWebKey> fetchOnce(String jwksURL);                                                                    // NEW
  public static List<JSONWebKey> fetchOnce(String jwksURL, Consumer<HttpURLConnection> customizer);                            // NEW
  public static List<JSONWebKey> fetchOnce(String jwksURL, FetchLimits limits, Consumer<HttpURLConnection> customizer);        // NEW
}
```

Internally, `fromIssuer(issuer)` and `fromWellKnown(url)` delegate to `OpenIDConnect.discover(issuer)` / `discoverFromWellKnown(url)` to obtain an `OpenIDConnectConfiguration`, then funnel into the same `fromConfiguration` code path. There is one implementation; the three convenience entry points exist so that callers who only want JWT verification do not have to write the two-step pattern themselves.

`fromJWKS(url)` skips discovery entirely — it requires a direct JWKS URL and never performs an OIDC fetch.

`of(...)` returns a `JWKS` directly because there is nothing to configure for a static set: no refresh interval, no clock, no scheduler, no HTTP customizer. The asymmetry between `of(...) → JWKS` and the other factories returning `Builder` is intentional and reflects the underlying truth.

`fetchOnce(...)` is the one-shot replacement for today's `JSONWebKeySetHelper.retrieveKeysFromJWKS(...)`. It does not construct a `JWKS` instance, does not schedule any background work, does not need `close()`. Callers who later want resolver semantics can pass the result to `JWKS.of(...)`.

### 3.2 Instance surface

```java
// VerifierResolver — existing
@Override public Verifier resolve(Header header);

// Raw JWK access — NEW
public JSONWebKey get(String kid);             // null if absent
public Collection<JSONWebKey> keys();          // unmodifiable, JWKS-endpoint order
public Set<String> keyIds();                   // unmodifiable, JWKS-endpoint order; renames currentKids()

// Operational — existing semantics, all no-op / null / 0 for JWKS.of(...)
public void refresh();
public int consecutiveFailures();
public Instant lastFailedRefresh();
public Instant lastSuccessfulRefresh();
public Instant nextDueAt();
@Override public void close();
```

`get(String)` was chosen over `byKid(String)` because `JWKS` is conceptually a kid-indexed key set; `get` matches the intuition without introducing a new word. `resolve(Header)` and `get(String)` look up against the same snapshot — a JWK and its derived `Verifier` are always present together (or both absent together).

`keys()` and `keyIds()` reflect JWKS-endpoint insertion order via `LinkedHashMap` / `LinkedHashSet`, preserving the ordering guarantee `currentKids()` provides today.

For `JWKS.of(...)`:
- `resolve` and `get` work identically to the remote case.
- `refresh()` is a no-op (returns normally — the snapshot is already complete).
- `consecutiveFailures()` returns 0.
- `lastFailedRefresh()`, `lastSuccessfulRefresh()`, `nextDueAt()` return null.
- `close()` is a no-op (no scheduler, no inflight worker, no thread to interrupt).

### 3.3 Snapshot

The internal `Snapshot` record gains a `Map<String, JSONWebKey> jwkByKid` alongside the existing `Map<String, Verifier> byKid`. Both maps populate together at fetch (and atomically swap together on refresh) so a `get(kid)` and a `resolve(headerWithKid)` against the same snapshot are guaranteed consistent. `JWKS.of(...)` constructs a snapshot once, sets it, and never rotates it.

### 3.4 Refresh behavior

Discovery happens **once** at construction, never on subsequent refreshes. The `OpenIDConnectConfiguration` is consulted to extract `jwks_uri` (and validate it is non-null and non-empty); subsequent refreshes hit `jwks_uri` directly via the same single-flight, snapshot-swap, exponential-backoff machinery `JWKSource` already implements. Re-discovery is not automatic — if a server rotates `jwks_uri`, the caller must rebuild the `JWKS`.

This is a behavior change from today's `JWKSource.fromIssuer` / `fromWellKnownConfiguration`, which re-fetch the discovery doc on every refresh. The change is a strict efficiency win for compliant servers (one HTTP request per refresh instead of two) and reflects observed real-world behavior: `jwks_uri` is stable for the lifetime of an issuer.

All other refresh semantics carry forward unchanged: `singleflightRefresh`, `failureSnapshot`, `chosenInterval` (Cache-Control honoring), `parseRetryAfter`, the virtual-thread worker, `CacheControlPolicy.CLAMP` / `IGNORE`, the `consecutiveFailures` watermark, and the `nextDueAt` floor.

### 3.5 Builder

The `JWKS.Builder` is the same shape as today's `JWKSource.Builder` plus one new method:

```java
public Builder fetchLimits(FetchLimits limits);   // NEW; default: FetchLimits.defaults()
```

All existing builder methods (`scheduledRefresh`, `refreshInterval`, `minRefreshInterval`, `refreshTimeout`, `refreshOnMiss`, `cacheControlPolicy`, `clock`, `logger`, `httpConnectionCustomizer`) carry forward unchanged.

**Builder-scope semantics for shared knobs.** When `fromIssuer(issuer)` or `fromWellKnown(url)` is used, the construction-time discovery hop performed by `Builder.build()` uses the Builder's `httpConnectionCustomizer` and `fetchLimits` (the same values that apply to subsequent JWKS refreshes). This matches today's `JWKSource` behavior, where `httpConnectionCustomizer` flows through `JSONWebKeySetHelper.retrieveKeysFromWellKnownConfiguration` to both the discovery and JWKS hops. When `fromConfiguration(cfg)` is used, no discovery hop is performed at build (the configuration is already in hand) and these knobs apply only to the JWKS hop.

**Build-time fetch semantics.** `Builder.build()` dispatches the initial fetch (discovery hop when applicable, then the JWKS hop) on the singleflight worker and blocks the calling thread up to `refreshTimeout` waiting for it. This matches today's `JWKSource` behavior: timing out, network failures, and parse failures during the initial fetch do **not** throw from `build()` — the returned `JWKS` carries an empty snapshot, `consecutiveFailures() ≥ 1`, and the async worker continues. Callers who want fail-fast semantics should call `refresh()` after `build()` and react to the `JWKSFetchException` it surfaces.

**`fromConfiguration` validation.** When `fromConfiguration(cfg)` is used, no discovery hop is performed; therefore the absence of `jwks_uri` is a synchronous, structural error rather than an async fetch failure. `Builder.build()` throws `IllegalArgumentException` if `cfg.jwksURI()` is null or empty, before any fetch is dispatched. This is new behavior (today there is no `fromConfiguration` factory).

### 3.6 `JWKSFetchException`

`JWKSRefreshException` is renamed to `JWKSFetchException`. The same `Reason` enum (`TIMEOUT`, `NON_2XX`, `NETWORK`, `PARSE`, `EMPTY_RESULT`) is thrown by:

- `refresh()` (synchronous refresh on a remote-backed `JWKS`)
- `fetchOnce(...)` (one-shot static fetch)
- The initial fetch performed inside `Builder.build()` when it fails

The "Refresh" suffix made sense when refresh was the only path; with `fetchOnce` and the construction-time fetch sharing the same failure modes, "Fetch" is the honest name.

`JSONWebKeySetException` (the inner class on `JSONWebKeySetHelper`) is removed — its callers either disappear with the helper's deletion or migrate to `JWKSFetchException`.

## 4. `FetchLimits`

A new top-level type at `org.lattejava.jwt.FetchLimits` replaces the volatile static config on `JSONWebKeySetHelper` and `ServerMetaDataHelper`.

```java
public final class FetchLimits {
  public static FetchLimits defaults();
  public static Builder builder();

  public int maxResponseBytes();          // default: 1 MiB
  public int maxRedirects();              // default: 3
  public int maxNestingDepth();           // default: 16
  public int maxNumberLength();           // default: 1000
  public int maxObjectMembers();          // default: 1000
  public int maxArrayElements();          // default: 10000
  public boolean allowDuplicateJSONKeys();// default: false

  public static final class Builder {
    public Builder maxResponseBytes(int n);
    public Builder maxRedirects(int n);
    public Builder maxNestingDepth(int n);
    public Builder maxNumberLength(int n);
    public Builder maxObjectMembers(int n);
    public Builder maxArrayElements(int n);
    public Builder allowDuplicateJSONKeys(boolean allow);
    public FetchLimits build();
  }
}
```

`FetchLimits` is immutable. Defaults exactly match today's `JSONWebKeySetHelper.DEFAULT_*` constants. Validation matches today's setters: every numeric limit must be strictly positive (the response cap and parser caps cannot be disabled).

`FetchLimits.defaults()` returns a singleton — callers that do not need to override anything do not allocate.

The same `FetchLimits` instance is consumed by `JWKS.Builder.fetchLimits(...)`, `JWKS.fetchOnce(..., FetchLimits, ...)`, and `OpenIDConnect.discover(..., FetchLimits, ...)` / `discoverFromWellKnown(..., FetchLimits, ...)`. Callers who need uniform limits across discovery + JWKS build a single `FetchLimits` once and reuse it.

The same hardening applies to discovery responses as to JWKS responses, intentionally. A discovery doc is a flat JSON object in spec, but we cannot trust the wire to honor that — a misconfigured or hostile server can return arbitrary payloads, and the cost of the limits is zero for compliant servers (real OIDC docs are well under all the defaults).

## 5. Removals

### 5.1 Deleted classes

| Symbol | Migration |
|---|---|
| `org.lattejava.jwt.oauth2.AuthorizationServerMetaData` | `OpenIDConnectConfiguration` (superset of fields; null OIDC-specific fields when fetching from an RFC 8414 endpoint) |
| `org.lattejava.jwt.oauth2.ServerMetaDataHelper` | `OpenIDConnect.discoverFromWellKnown(url)` against the OAuth or OIDC well-known URL; or fetch JSON yourself and map manually |
| `org.lattejava.jwt.jwks.JSONWebKeySetHelper` | See §5.2 below — pieces redistribute to `JWKS`, `OpenIDConnect`, and `internal/` |
| `org.lattejava.jwt.jwks.JSONWebKeySetHelper.JSONWebKeySetException` | `JWKSFetchException` |

### 5.2 `JSONWebKeySetHelper` redistribution

The helper class is deleted entirely. Its current responsibilities split as follows:

| Today | New home |
|---|---|
| `retrieveKeysFromIssuer(...)` overloads | Removed. Use `OpenIDConnect.discover(issuer)` for the doc, and/or `JWKS.fromIssuer(issuer).build()` for a lifecycle-managed JWKS. |
| `retrieveKeysFromWellKnownConfiguration(...)` overloads | Removed. Use `OpenIDConnect.discoverFromWellKnown(url)` for the doc, and/or `JWKS.fromWellKnown(url).build()` for a lifecycle-managed JWKS. |
| `retrieveKeysFromJWKS(String[, Consumer])` | `JWKS.fetchOnce(url[, customizer][, limits])` |
| `retrieveKeysFromJWKS(HttpURLConnection)` | Removed. Constructing your own `HttpURLConnection` is no longer a supported entry point — use `JWKS.fetchOnce` with a `Consumer<HttpURLConnection>` customizer to reach the same configuration. |
| `parseJSON(InputStream)` | Moved to `org.lattejava.jwt.internal.HardenedJSON.parse(InputStream, FetchLimits)` (package-private to `internal/`). Used by `JWKS` and `OpenIDConnect` internals. |
| `setMaxResponseSize`, `setMaxRedirects`, `setMaxNestingDepth`, `setMaxNumberLength`, `setMaxObjectMembers`, `setMaxArrayElements`, `setAllowDuplicateJSONKeys`, `resetDefaults`, and the matching `DEFAULT_*` constants | Removed. Configure via `FetchLimits` per-instance (`JWKS.Builder.fetchLimits(...)`) or per-call (`fetchOnce(..., limits, ...)`, `discover(..., limits, ...)`). |
| The package-private `retrieveJWKSResponseFrom*` triplet (used by `JWKSource`) | Collapses into a single private fetch path inside `JWKS` itself; no longer needs to live in a separate helper. |

### 5.3 Deleted package

`org.lattejava.jwt.oauth2` is empty after the above and is removed.

## 6. 7.0 changelog

Recommended changelog entry text:

> **Discovery + JWKS surface** (breaking)
>
> - **New** `org.lattejava.jwt.OpenIDConnectConfiguration` POJO models the full OIDC Discovery 1.0 Provider Metadata superset (plus RFC 8414 fields and routinely-deployed OIDC extensions).
> - **New** `OpenIDConnect.discover(issuer)` and `OpenIDConnect.discoverFromWellKnown(url)` are the entry points for fetching the configuration. Both also accept an optional `FetchLimits` and `Consumer<HttpURLConnection>` customizer.
> - **New** `org.lattejava.jwt.FetchLimits` carries hardening limits (response size, redirects, JSON parser caps) per-instance / per-call. Replaces the volatile static config on `JSONWebKeySetHelper` and `ServerMetaDataHelper`.
> - `JWKSource` is renamed to `JWKS`. Adds `get(String kid)`, `keys()`, `keyIds()` (renamed from `currentKids()`), `fromConfiguration(OpenIDConnectConfiguration)`, `of(List<JSONWebKey>)` / `of(JSONWebKey...)`, and `fetchOnce(...)`. Adds `Builder.fetchLimits(...)`.
> - `JWKSRefreshException` is renamed to `JWKSFetchException`.
> - `JWKS.fromIssuer` / `JWKS.fromWellKnown` now perform discovery exactly once at construction and cache `jwks_uri`; subsequent refreshes hit the JWKS endpoint directly instead of re-discovering.
> - **Removed:** `org.lattejava.jwt.oauth2.AuthorizationServerMetaData`, `org.lattejava.jwt.oauth2.ServerMetaDataHelper`, the entire `org.lattejava.jwt.oauth2` package, and `org.lattejava.jwt.jwks.JSONWebKeySetHelper`. Migration paths above.

## 7. Out of scope / future work

- **Automatic re-discovery.** If a server rotates `jwks_uri` between configuration fetches, `JWKS` will not detect this — it will keep refreshing against the cached URL and (eventually) fail. A future spec may add an opt-in re-discovery trigger (e.g., re-discover on N consecutive `NON_2XX` failures, or on an explicit signal). Not in 7.0.
- **Async / non-blocking discovery.** `OpenIDConnect.discover(...)` is synchronous and blocking. A future overload returning `CompletableFuture<OpenIDConnectConfiguration>` may be added if a use case emerges.
- **Streaming fetch hooks.** `JWKS.fetchOnce` returns a fully-materialized list. There is no streaming variant.
- **`OpenIDConnectConfiguration.discover()` instance method.** Discovery is a static operation that *returns* the configuration; chaining a method off the type that does not yet exist is an awkward shape. The static entry points on `OpenIDConnect` are the supported front door.
