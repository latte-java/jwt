# Discovery + JWKS simplification

| Field | Value |
|------|------|
| Status | Implemented |
| Revision | 3 |
| Last updated | 2026-04-26 |
| Owner | Daniel DeGroff |
| Target version | 7.0 (breaking) |

## Problem statement

The library presents three concepts to users — **JWT**, **JWKS**, and **OpenID Connect Discovery** — but only two of them are first-class types today. OIDC discovery is implicit: it happens inside `JSONWebKeySetHelper` when the caller asks for keys by issuer or by well-known URL, and the helper extracts `jwks_uri` and discards everything else. `JWKSource` re-does the same hop on every refresh.

There is an `AuthorizationServerMetaData` POJO sitting in `org.lattejava.jwt.oauth2` that models RFC 8414 (a strict OAuth 2.0 subset of OIDC discovery), with a sibling `ServerMetaDataHelper` to fetch it. Neither is wired into `JWKSource`, neither covers the OIDC superset, and the OAuth-only metadata they expose is out of scope for a JWT library.

The result: a caller who wants the OIDC configuration *and* JWKS fetching has to either (a) hand-walk discovery to extract `jwks_uri` themselves, then call `JWKSource.fromJWKS(...)`, or (b) call `JWKSource.fromIssuer(...)` and accept that the discovery doc is fetched, parsed, and thrown away on every refresh.

Naming compounds the friction: `JWKSource` means "self-refreshing remote-backed JWK set with a `VerifierResolver` interface" — but the spec word for "JSON Web Key Set" is `JWKS`. Calling the class `JWKS` matches both the spec and the user's mental model, and lets the same object serve as a simple `kid → JSONWebKey` lookup in addition to its resolver behavior.

## Goals

1. Promote OIDC discovery to a first-class type: `OpenIDConnectConfiguration`, fetched via `OpenIDConnect.discover(...)` / `discoverFromWellKnown(...)`. Useful as a standalone object outside JWT verification.
2. Rename `JWKSource` to `JWKS`; add raw-JWK lookup (`get(String)`, `keys()`, `keyIds()`) and a `fromConfiguration(OpenIDConnectConfiguration)` factory; add a static, non-refreshing `JWKS.of(...)` factory.
3. Collapse the standalone JWKS / discovery helpers into the new types. One class to look at when finding a JWKS or a discovery operation.
4. Replace process-global hardening config with per-instance / per-call `FetchLimits`. No more global mutable state.
5. Carry forward today's hardening defaults exactly (limits, response cap, redirect cap, JSON parse caps).

## Non-goals

- No changes to `JWT`, `Header`, `JWTDecoder`, `JWTEncoder`, the `Signer` / `Verifier` family, the algorithm packages, or the zero-dep stance.
- No automatic re-discovery if `jwks_uri` changes server-side. Discovery is one-shot at construction time. Callers who know the server moved must rebuild.
- No support for the RFC 8414 OAuth-only metadata endpoint (`/.well-known/oauth-authorization-server`) as a first-class operation. Callers with pure-OAuth servers fetch the JSON themselves and feed `jwks_uri` into `JWKS.fromJWKS(...)`.

## 1. `OpenIDConnectConfiguration`

A new top-level immutable POJO at `org.lattejava.jwt.OpenIDConnectConfiguration`.

### 1.1 Shape

Models the commonly-deployed subset of OpenID Connect Discovery 1.0 Provider Metadata and RFC 8414 Authorization Server Metadata. Every typed field is nullable — a pure-OAuth response will have null OIDC-specific fields, and an OIDC response without RFC 8414 extras will have null introspection / revocation fields.

The typed surface is deliberately a subset, not a superset, of the full metadata. Promoting an `otherClaims()` key to a typed accessor in a future 7.x release is a non-breaking addition (it adds a method); demoting a typed accessor is breaking. The 7.0 cut therefore reserves typed accessors for fields the library or its callers actually consume during JWT verification or routine OIDC client work, and lets everything else live in `otherClaims()`.

Typed accessors:

| Source | Fields |
|---|---|
| OIDC Core (REQUIRED) | `issuer`, `authorizationEndpoint`, `tokenEndpoint`, `jwksURI`, `responseTypesSupported`, `subjectTypesSupported`, `idTokenSigningAlgValuesSupported` |
| OIDC Core (RECOMMENDED) | `userinfoEndpoint`, `registrationEndpoint`, `scopesSupported`, `claimsSupported` |
| Routinely deployed | `grantTypesSupported`, `responseModesSupported`, `acrValuesSupported`, `codeChallengeMethodsSupported`, `tokenEndpointAuthMethodsSupported`, `tokenEndpointAuthSigningAlgValuesSupported`, `requestParameterSupported`, `requestURIParameterSupported`, `requireRequestURIRegistration` |
| OIDC extensions / RFC 8414 | `endSessionEndpoint` (RP-Initiated Logout 1.0), `introspectionEndpoint`, `revocationEndpoint` |

Everything else lands in `otherClaims() : Map<String, Object>` and is preserved through round-trip serialization. Notably **not** typed in 7.0:

- All ID token / userinfo / request object encryption fields (`id_token_encryption_*`, `userinfo_encryption_*`, `request_object_encryption_*`, plus the `*_signing_alg_values_supported` siblings for userinfo and request objects). These are extension territory and almost never consumed by JWT-verification code.
- The i18n / UX fields: `display_values_supported`, `claim_types_supported`, `claims_locales_supported`, `ui_locales_supported`, `claims_parameter_supported`.
- The informational links: `service_documentation`, `op_policy_uri`, `op_tos_uri`.
- Session Management 1.0 (`check_session_iframe`) and Front-/Back-Channel Logout (`frontchannel_logout_*`, `backchannel_logout_*`). Deployed unevenly; users who need them read from `otherClaims()`.
- The RFC 8414 introspection / revocation auth-method and signing-alg sublists (`introspection_endpoint_auth_methods_supported` and friends). Resource-server territory, not verification territory.

Acronym casing follows the project rule: `jwksURI()`, `requestURIParameterSupported()`, `requireRequestURIRegistration()`. (The existing `AuthorizationServerMetaData` had `jwksUri()` / `opPolicyUri()` / `opTosUri()`, which violated the rule; the new POJO fixes this for the typed fields it carries forward.)

### 1.2 Construction and serialization

```java
public final class OpenIDConnectConfiguration {
  // Builder is the only public path to construction; constructor is private.
  public static Builder builder();

  // Round-trip back to a JSON-serializable Map.
  public Map<String, Object> toSerializableMap();

  public String toJSON();          // serializes via LatteJSONProcessor
  @Override public String toString();   // == toJSON()
  @Override public boolean equals(Object o);
  @Override public int hashCode();
}
```

The builder is reusable — every builder in this library is reusable unless it has a specific reason not to be, and `OpenIDConnectConfiguration.Builder` follows the convention. Each `build()` returns a fresh immutable instance with independent collection copies. List-typed accessors and `otherClaims()` return unmodifiable views, so the immutability claim holds even when the caller retains a reference to a returned collection.

`toSerializableMap()` flattens `otherClaims()` entries back to top-level keys alongside the typed fields (e.g. an Auth0 `mfa_challenge_endpoint` that landed in `otherClaims` round-trips to a top-level `mfa_challenge_endpoint` in the output map). A typed field and an `otherClaims` entry cannot share a key — `fromMap`-style ingestion routes a recognized key to its typed accessor and only unrecognized keys to `otherClaims`.

**Null handling on serialization.** `toSerializableMap()` and `toJSON()` omit typed fields whose value is null; absent fields stay absent on the wire rather than emitting `"key": null`. Round-trip semantics: a field absent in the input doc lands as null on the typed accessor, and a null typed field is absent on output. JSON-null in the input is treated identically to absent — the wire shouldn't be sending `"jwks_uri": null` and the library does not invent a distinction.

**`toString()`.** Returns `toJSON()`, matching the existing convention in `JWT` and other top-level immutable POJOs in the library.

There is no public `fromMap(Map)` factory. Applications that have already parsed a discovery document with their own JSON library construct a configuration via `builder()`. The library's own discovery path (`OpenIDConnect.discover` / `discoverFromWellKnown`) does the parse → typed-field routing internally and is the only supported network path.

### 1.3 Exception type

A new `org.lattejava.jwt.OpenIDConnectException extends RuntimeException` covers all discovery-fetch failures (network, non-2xx, JSON parse, missing `jwks_uri`, missing `issuer`, oversize response, redirect overflow, etc.). It does **not** extend `JWTException` — discovery is not a JWT operation, and putting it under `JWTException` would mislead `catch` blocks that target JWT-specific failures.

## 2. `OpenIDConnect.discover` entry points

Added to the existing `org.lattejava.jwt.OpenIDConnect` class, alongside the existing `at_hash` / `c_hash` static helpers (no rename, no removal).

```java
public static OpenIDConnectConfiguration discover(String issuer);
public static OpenIDConnectConfiguration discover(String issuer, Consumer<HttpURLConnection> customizer);
public static OpenIDConnectConfiguration discover(String issuer, FetchLimits limits);
public static OpenIDConnectConfiguration discover(String issuer, FetchLimits limits, Consumer<HttpURLConnection> customizer);

public static OpenIDConnectConfiguration discoverFromWellKnown(String wellKnownURL);
public static OpenIDConnectConfiguration discoverFromWellKnown(String wellKnownURL, Consumer<HttpURLConnection> customizer);
public static OpenIDConnectConfiguration discoverFromWellKnown(String wellKnownURL, FetchLimits limits);
public static OpenIDConnectConfiguration discoverFromWellKnown(String wellKnownURL, FetchLimits limits, Consumer<HttpURLConnection> customizer);
```

`discover(issuer)` trims a trailing slash on the issuer and appends `/.well-known/openid-configuration` (matches today's `JSONWebKeySetHelper.retrieveKeysFromIssuer` behavior). It also enforces OIDC Discovery 1.0 §4.3: the `issuer` field in the response must equal the issuer argument, and a mismatch raises `OpenIDConnectException`. This guards against DNS hijack / CDN takeover scenarios where a substituted discovery doc would otherwise feed attacker-controlled metadata into the verifier.

**Trailing-slash normalization.** The equality check is performed after trimming a single trailing slash from both sides (input issuer and the response `issuer` field), then comparing for byte-identical equality. This is a deliberate deviation from a strict reading of RFC §4.3, made because real-world servers vary in trailing-slash treatment and the hijacking property the RFC is protecting (different host or path) is unaffected by slash normalization. Other implementations (Nimbus, Auth0 java-jwt) normalize similarly in practice.

`discoverFromWellKnown(url)` takes a fully-qualified URL — the same overload is the right entry point for an RFC 8414 server's `/.well-known/oauth-authorization-server` URL, since the JSON shape is a subset and every OIDC-specific accessor on `OpenIDConnectConfiguration` simply returns null. This overload deliberately does **not** perform issuer-equality validation (no expected issuer is supplied), and the Javadoc must call this out as a security downgrade relative to `discover(issuer)`. Callers with an OIDC issuer should prefer `discover(issuer)`.

The `Consumer<HttpURLConnection>` is the same customizer hook today's helpers expose (auth headers, custom timeouts, proxies). Omitting it leaves the connection at its defaults.

When `FetchLimits` is omitted, `FetchLimits.defaults()` applies. See §4.

A discovery call performs exactly one HTTP request. The response is parsed once, fields land in the typed accessors of `OpenIDConnectConfiguration`, and the object is returned. No caching is layered into this method — the caller decides how long to hold the configuration. (`JWKS.fromConfiguration(cfg)` accepts the same configuration directly.)

## 3. `JWKS` (replaces `JWKSource`)

Renamed type at `org.lattejava.jwt.jwks.JWKS`. Implements `VerifierResolver` and `AutoCloseable`, same as today's `JWKSource`.

### 3.1 Static factories

```java
public final class JWKS implements VerifierResolver, AutoCloseable {
  // Remote-backed, lifecycle-managed
  public static Builder fromIssuer(String issuer);
  public static Builder fromWellKnown(String wellKnownURL);                 // renamed from fromWellKnownConfiguration
  public static Builder fromJWKS(String jwksURL);
  public static Builder fromConfiguration(OpenIDConnectConfiguration cfg);  // NEW

  // Static, non-refreshing — no Builder
  public static JWKS of(List<JSONWebKey> keys);                             // NEW
  public static JWKS of(JSONWebKey... keys);                                // NEW

  // One-shot fetch — no instance, no lifecycle
  public static List<JSONWebKey> fetchOnce(String jwksURL);                                                                    // NEW
  public static List<JSONWebKey> fetchOnce(String jwksURL, Consumer<HttpURLConnection> customizer);                            // NEW
  public static List<JSONWebKey> fetchOnce(String jwksURL, FetchLimits limits);                                                // NEW
  public static List<JSONWebKey> fetchOnce(String jwksURL, FetchLimits limits, Consumer<HttpURLConnection> customizer);        // NEW
}
```

Internally, `fromIssuer(issuer)` and `fromWellKnown(url)` delegate to `OpenIDConnect.discover(issuer)` / `discoverFromWellKnown(url)` to obtain an `OpenIDConnectConfiguration`, then funnel into the same `fromConfiguration` code path. There is one implementation; the three convenience entry points exist so that callers who only want JWT verification do not have to write the two-step pattern themselves.

`fromJWKS(url)` skips discovery entirely — it requires a direct JWKS URL and never performs an OIDC fetch.

`of(...)` returns a `JWKS` directly because there is nothing to configure for a static set: no refresh interval, no clock, no scheduler, no HTTP customizer. The asymmetry between `of(...) → JWKS` and the other factories returning `Builder` is intentional and reflects the underlying truth.

`fetchOnce(...)` is the one-shot replacement for today's `JSONWebKeySetHelper.retrieveKeysFromJWKS(...)`. It does not construct a `JWKS` instance, does not schedule any background work, does not need `close()`. Callers who later want resolver semantics can pass the result to `JWKS.of(...)`.

### 3.2 Instance surface

```java
// VerifierResolver — existing
@Override public Verifier resolve(Header header);

// Raw JWK access — NEW
public JSONWebKey get(String kid);             // null if absent
public Collection<JSONWebKey> keys();          // unmodifiable, JWKS-endpoint order
public Set<String> keyIds();                   // unmodifiable, JWKS-endpoint order; renames currentKids()

// Operational — existing semantics, all no-op / null / 0 for JWKS.of(...)
public void refresh();
public int consecutiveFailures();
public Instant lastFailedRefresh();
public Instant lastSuccessfulRefresh();
public Instant nextDueAt();
@Override public void close();
```

`get(String)` was chosen over `byKid(String)` because `JWKS` is conceptually a kid-indexed key set; `get` matches the intuition without introducing a new word. `resolve(Header)` and `get(String)` look up against the same snapshot — a JWK and its derived `Verifier` are always present together (or both absent together).

`keys()` and `keyIds()` reflect JWKS-endpoint insertion order via `LinkedHashMap` / `LinkedHashSet`, preserving the ordering guarantee `currentKids()` provides today.

**JWKs without a `kid`.** A `JSONWebKey` with a null `kid` is permitted in the snapshot. It appears in `keys()` (full insertion order) but is not indexed in the kid-keyed map: `get(null)` returns null and `resolve(header)` cannot match it (a header without a `kid` already returns null today, unchanged). The same rule applies to a snapshot built from a remote JWKS — kidless JWKs are visible via `keys()` but not resolvable. `keyIds()` excludes the null entry.

**Duplicate `kid`.** First-write-wins for both the kid-keyed map and `keys()` — at most one entry per kid lands in the snapshot. Subsequent JWKs sharing a kid are dropped with a WARN log entry (matches today's `JWKSource` behavior).

For `JWKS.of(...)`:
- `resolve` and `get` work identically to the remote case.
- `refresh()` is a no-op (returns normally — the snapshot is already complete).
- `consecutiveFailures()` returns 0.
- `lastFailedRefresh()`, `lastSuccessfulRefresh()`, `nextDueAt()` return null.
- `close()` is a no-op (no scheduler, no inflight worker, no thread to interrupt).
- `JWKS.of()` with no keys (or `of(List.of())`) is permitted, returns a non-null instance, and is not rejected at construction. `keys()` and `keyIds()` return empty collections; `get(kid)` returns null for any input; `resolve(header)` raises `MissingVerifierException` for any header. This is the same behavior as a remote-backed `JWKS` whose snapshot happens to be empty.

### 3.3 Snapshot

The internal `Snapshot` record gains a `Map<String, JSONWebKey> jwkByKid` alongside the existing `Map<String, Verifier> byKid`. Both maps populate together at fetch (and atomically swap together on refresh) so a `get(kid)` and a `resolve(headerWithKid)` against the same snapshot are guaranteed consistent. `JWKS.of(...)` constructs a snapshot once, sets it, and never rotates it.

### 3.4 Refresh behavior

Discovery happens **at most once successfully** over the lifetime of a `JWKS`, then never again. Concretely:

- The first refresh (the construction-time fetch dispatched inside `Builder.build()`) runs discovery, extracts and caches `jwks_uri`, and runs the JWKS hop.
- If discovery succeeds, `jwks_uri` is locked for the lifetime of the `JWKS` and every subsequent refresh hits it directly. No re-discovery.
- If discovery has not yet succeeded — because the construction-time discovery hop failed, or every refresh since has failed before reaching the JWKS hop — every refresh re-attempts discovery. Once one succeeds, the rule above kicks in.

This refines the spec language and removes the "what does refresh() do on a JWKS that never discovered?" gap: the answer is "it tries discovery again." Once discovery has succeeded, re-discovery is not automatic — if a server rotates `jwks_uri` after the lock, the caller must rebuild the `JWKS`.

This is a behavior change from today's `JWKSource.fromIssuer` / `fromWellKnownConfiguration`, which re-fetch the discovery doc on every refresh. The change is a strict efficiency win for compliant servers (one HTTP request per refresh instead of two, post-discovery) and reflects observed real-world behavior: `jwks_uri` is stable for the lifetime of an issuer.

All other refresh semantics carry forward unchanged: `singleflightRefresh`, `failureSnapshot`, `chosenInterval` (Cache-Control honoring), `parseRetryAfter`, the virtual-thread worker, `CacheControlPolicy.CLAMP` / `IGNORE`, the `consecutiveFailures` watermark, and the `nextDueAt` floor.

**Timeout scope.** `refreshTimeout` bounds *only* the JWKS-endpoint hop. The discovery hop is bounded by the `HttpURLConnection` connect/read timeouts (10s/10s default from `AbstractHTTPHelper.buildURLConnection`, tunable via the `Consumer<HttpURLConnection>` customizer). The asymmetry is intentional: `refreshTimeout` is conceptually about how long the caller is willing to wait *for keys*, not for the precursor metadata that names the key endpoint. Worst-case `build()` wall time = HTTP-connect + HTTP-read for discovery + `refreshTimeout` for JWKS.

### 3.5 Builder

The `JWKS.Builder` is the same shape as today's `JWKSource.Builder` plus two new methods:

```java
public Builder failFast(boolean failFast);       // NEW; default: false
public Builder fetchLimits(FetchLimits limits);  // NEW; default: FetchLimits.defaults()
```

All existing builder methods (`scheduledRefresh`, `refreshInterval`, `minRefreshInterval`, `refreshTimeout`, `refreshOnMiss`, `cacheControlPolicy`, `clock`, `logger`, `httpConnectionCustomizer`) carry forward unchanged. The builder is reusable; calling `build()` twice returns two independent `JWKS` instances.

**Builder-scope semantics for shared knobs.** When `fromIssuer(issuer)` or `fromWellKnown(url)` is used, the construction-time discovery hop performed by `Builder.build()` uses the Builder's `httpConnectionCustomizer` and `fetchLimits` (the same values that apply to subsequent JWKS refreshes). This matches today's `JWKSource` behavior, where `httpConnectionCustomizer` flows through `JSONWebKeySetHelper.retrieveKeysFromWellKnownConfiguration` to both the discovery and JWKS hops. When `fromConfiguration(cfg)` is used, no discovery hop is performed at build (the configuration is already in hand) and these knobs apply only to the JWKS hop.

**Build-time fetch semantics (default, `failFast == false`).** `Builder.build()` dispatches the initial fetch on the singleflight worker and blocks the calling thread up to `refreshTimeout` waiting for the JWKS hop. The initial fetch is the discovery hop followed by the JWKS hop for `fromIssuer` / `fromWellKnown`, and the JWKS hop only for `fromConfiguration` and `fromJWKS`. Timing out, network failures, and parse failures during the initial fetch do **not** throw from `build()` — the returned `JWKS` carries an empty snapshot, `consecutiveFailures() ≥ 1`, and the async worker continues. This matches today's `JWKSource` behavior. Callers can poll `lastSuccessfulRefresh()` (`null` ⇒ no fetch has succeeded yet) or `consecutiveFailures()` to assess health, or call `refresh()` and react to the exception it surfaces.

**Build-time fetch semantics (`failFast == true`).** The same initial fetch is dispatched, but if it fails, `build()` re-raises the underlying exception synchronously: `OpenIDConnectException` if discovery failed, `JWKSFetchException` if the JWKS hop failed. The async worker is shut down before the exception is thrown — a failed `failFast` build does not leak background threads. The default remains `false` because most callers prefer "the JWKS will heal itself when the issuer comes back" over "my service crashed because the issuer was briefly unreachable at boot."

**Health checks.** No new accessor is added. Callers wanting "is this `JWKS` ready?" use `lastSuccessfulRefresh() != null`. Callers wanting "has it ever failed?" use `consecutiveFailures() > 0` or `lastFailedRefresh() != null`. The existing surface already answers these questions.

**`fromConfiguration` validation.** When `fromConfiguration(cfg)` is used, no discovery hop is performed; therefore the absence of `jwks_uri` is a synchronous, structural error rather than an async fetch failure. `Builder.build()` throws `IllegalArgumentException` if `cfg.jwksURI()` is null or empty, before any fetch is dispatched. This is new behavior (today there is no `fromConfiguration` factory) and is independent of `failFast`.

### 3.6 `JWKSFetchException`

`JWKSRefreshException` is renamed to `JWKSFetchException`. The same `Reason` enum (`TIMEOUT`, `NON_2XX`, `NETWORK`, `PARSE`, `EMPTY_RESULT`) is thrown by:

- `refresh()` (synchronous refresh on a remote-backed `JWKS`) — *for the JWKS hop*
- `fetchOnce(...)` (one-shot static fetch)
- The initial fetch performed inside `Builder.build()` when `failFast == true` and the JWKS hop fails

The "Refresh" suffix made sense when refresh was the only path; with `fetchOnce` and the construction-time fetch sharing the same failure modes, "Fetch" is the honest name.

**No exception wrapping across discovery and JWKS.** Discovery failures throw `OpenIDConnectException` (and only that). JWKS-endpoint failures throw `JWKSFetchException` (and only that). Neither type wraps the other. The two failure surfaces remain disjoint, so callers can branch on type instead of walking a cause chain.

This has one user-visible consequence: `refresh()` on a remote-backed `JWKS` can throw *either* `OpenIDConnectException` or `JWKSFetchException`, depending on which phase failed. While discovery has not yet succeeded, refresh re-attempts discovery and surfaces `OpenIDConnectException` on failure (per §3.4). After discovery has locked `jwks_uri`, refresh runs only the JWKS hop and surfaces `JWKSFetchException`. Both extend `RuntimeException`, so callers who catch the parent are unaffected; callers who want phase-level dispatch catch each. The `refresh()` Javadoc must call this out explicitly.

`Builder.build()` follows the same rule when `failFast == true`: a discovery-hop failure throws `OpenIDConnectException`, a JWKS-hop failure throws `JWKSFetchException`.

`JSONWebKeySetException` (the inner class on `JSONWebKeySetHelper`) is removed — its callers either disappear with the helper's deletion or migrate to `JWKSFetchException`.

### 3.7 Redirect policy

Redirects are followed up to `FetchLimits.maxRedirects()` (default 3). By default they are confined to the same origin: an origin is the (scheme, host, port) tuple of the request URL, and a redirect that changes any of the three is rejected. This applies uniformly to the discovery hop and the JWKS hop. Callers who need cross-origin redirects opt in via `FetchLimits.allowCrossOriginRedirects(true)` (default `false`).

The rationale: real OIDC providers rarely depend on cross-origin redirects mid-fetch — if they want JWKS hosted on a different host, they put that host directly in `jwks_uri` (which is exactly what the discovery doc is for). Cross-origin redirects, by contrast, are the highest-leverage attack surface: a DNS hijack or CDN takeover targeting the issuer host can silently swap the verifier's keys via a 302 to attacker-controlled infrastructure. Blocking by default costs nothing for compliant servers and removes an entire class of attack.

The opt-in exists for the long tail of enterprise SSO deployments behind gateways that do legitimately redirect across origins. Setting `allowCrossOriginRedirects(true)` is a deliberate, documented security trade-off — the Javadoc on the setter must say so.

A rejected cross-origin redirect raises `JWKSFetchException` (or `OpenIDConnectException` from a direct `discover` call) with a reason equivalent to today's `NETWORK`. The exception message identifies the rejected hop and names both origins (source and target) so operators can self-diagnose.

## 4. `FetchLimits`

A new top-level type at `org.lattejava.jwt.FetchLimits` replaces the volatile static config on `JSONWebKeySetHelper` and `ServerMetaDataHelper`.

```java
public final class FetchLimits {
  public static FetchLimits defaults();
  public static Builder builder();

  public int maxResponseBytes();             // default: 1 MiB
  public int maxRedirects();                 // default: 3
  public int maxNestingDepth();              // default: 16
  public int maxNumberLength();              // default: 1000
  public int maxObjectMembers();             // default: 1000
  public int maxArrayElements();             // default: 10000
  public boolean allowDuplicateJSONKeys();   // default: false
  public boolean allowCrossOriginRedirects();// default: false

  public static final class Builder {
    public Builder allowCrossOriginRedirects(boolean allow);
    public Builder allowDuplicateJSONKeys(boolean allow);
    public Builder maxArrayElements(int n);
    public Builder maxNestingDepth(int n);
    public Builder maxNumberLength(int n);
    public Builder maxObjectMembers(int n);
    public Builder maxRedirects(int n);
    public Builder maxResponseBytes(int n);
    public FetchLimits build();
  }
}
```

`FetchLimits` is immutable. Defaults exactly match today's `JSONWebKeySetHelper.DEFAULT_*` constants for the carried-forward fields; `allowCrossOriginRedirects` is new and defaults to `false` (stricter than today, which has no origin check). Validation matches today's setters: every numeric limit must be strictly positive (the response cap and parser caps cannot be disabled).

`FetchLimits.defaults()` returns a singleton — callers that do not need to override anything do not allocate. The builder is reusable.

The same `FetchLimits` instance is consumed by `JWKS.Builder.fetchLimits(...)`, `JWKS.fetchOnce(..., FetchLimits, ...)`, and `OpenIDConnect.discover(..., FetchLimits, ...)` / `discoverFromWellKnown(..., FetchLimits, ...)`. Callers who need uniform limits across discovery + JWKS build a single `FetchLimits` once and reuse it.

The same hardening applies to discovery responses as to JWKS responses, intentionally. A discovery doc is a flat JSON object in spec, but we cannot trust the wire to honor that — a misconfigured or hostile server can return arbitrary payloads, and the cost of the limits is zero for compliant servers (real OIDC docs are well under all the defaults).

## 5. Removals

### 5.1 Deleted classes

| Symbol | Migration |
|---|---|
| `org.lattejava.jwt.oauth2.AuthorizationServerMetaData` | `OpenIDConnectConfiguration` (superset of fields; null OIDC-specific fields when fetching from an RFC 8414 endpoint) |
| `org.lattejava.jwt.oauth2.ServerMetaDataHelper` | `OpenIDConnect.discoverFromWellKnown(url)` against the OAuth or OIDC well-known URL; or fetch JSON yourself and map manually |
| `org.lattejava.jwt.jwks.JSONWebKeySetHelper` | See §5.2 below — pieces redistribute to `JWKS`, `OpenIDConnect`, and `internal/` |
| `org.lattejava.jwt.jwks.JSONWebKeySetHelper.JSONWebKeySetException` | `JWKSFetchException` |

### 5.2 `JSONWebKeySetHelper` redistribution

The helper class is deleted entirely. Its current responsibilities split as follows:

| Today | New home |
|---|---|
| `retrieveKeysFromIssuer(...)` overloads | Removed. Use `OpenIDConnect.discover(issuer)` for the doc, and/or `JWKS.fromIssuer(issuer).build()` for a lifecycle-managed JWKS. |
| `retrieveKeysFromWellKnownConfiguration(...)` overloads | Removed. Use `OpenIDConnect.discoverFromWellKnown(url)` for the doc, and/or `JWKS.fromWellKnown(url).build()` for a lifecycle-managed JWKS. |
| `retrieveKeysFromJWKS(String[, Consumer])` | `JWKS.fetchOnce(url[, customizer][, limits])` |
| `retrieveKeysFromJWKS(HttpURLConnection)` | Removed. Constructing your own `HttpURLConnection` is no longer a supported entry point — use `JWKS.fetchOnce` with a `Consumer<HttpURLConnection>` customizer to reach the same configuration. |
| `parseJSON(InputStream)` | Moved to `org.lattejava.jwt.internal.HardenedJSON.parse(InputStream, FetchLimits)` (package-private to `internal/`). Used by `JWKS` and `OpenIDConnect` internals. |
| `setMaxResponseSize`, `setMaxRedirects`, `setMaxNestingDepth`, `setMaxNumberLength`, `setMaxObjectMembers`, `setMaxArrayElements`, `setAllowDuplicateJSONKeys`, `resetDefaults`, and the matching `DEFAULT_*` constants | Removed. Configure via `FetchLimits` per-instance (`JWKS.Builder.fetchLimits(...)`) or per-call (`fetchOnce(..., limits, ...)`, `discover(..., limits, ...)`). |
| The package-private `retrieveJWKSResponseFrom*` triplet (used by `JWKSource`) | Collapses into a single private fetch path inside `JWKS` itself; no longer needs to live in a separate helper. |

### 5.3 Deleted package

`org.lattejava.jwt.oauth2` is empty after the above and is removed. The `exports org.lattejava.jwt.oauth2;` clause in `src/main/java/module-info.java` is removed alongside the package. Downstream module consumers that `requires org.lattejava.jwt;` will see the type rename as a compile error rather than a missing-package error, but the migration path in §5.1 covers it.

## 6. 7.0 changelog

Recommended changelog entry text:

> **Discovery + JWKS surface** (breaking)
>
> - **New** `org.lattejava.jwt.OpenIDConnectConfiguration` POJO models a typed subset of OIDC Discovery 1.0 Provider Metadata and RFC 8414 Authorization Server Metadata. Fields outside the typed surface land in `otherClaims()` and round-trip through serialization. The typed cut is a deliberate subset; promoting an `otherClaims` key in a future 7.x release is non-breaking.
> - **New** `OpenIDConnect.discover(issuer)` and `OpenIDConnect.discoverFromWellKnown(url)` are the entry points for fetching the configuration. Both also accept an optional `FetchLimits` and `Consumer<HttpURLConnection>` customizer. `discover(issuer)` enforces OIDC Discovery 1.0 §4.3 issuer-equality validation (with a single-trailing-slash normalization on both sides); `discoverFromWellKnown(url)` does not (it has no expected issuer to compare against) and is documented as a security downgrade relative to `discover(issuer)`.
> - **New** `org.lattejava.jwt.FetchLimits` carries hardening limits (response size, redirects, JSON parser caps, cross-origin redirect opt-in) per-instance / per-call. Replaces the volatile static config on `JSONWebKeySetHelper` and `ServerMetaDataHelper`.
> - **Hardening:** redirects are now confined to the same origin (scheme + host + port) by default on both the discovery and JWKS hops. Cross-origin redirects are rejected unless `FetchLimits.allowCrossOriginRedirects(true)` is set. (Previously: redirects followed up to `maxRedirects` regardless of origin.)
> - `JWKSource` is renamed to `JWKS`. Adds `get(String kid)`, `keys()`, `keyIds()` (renamed from `currentKids()`), `fromConfiguration(OpenIDConnectConfiguration)`, `of(List<JSONWebKey>)` / `of(JSONWebKey...)`, and `fetchOnce(...)`. Adds `Builder.fetchLimits(...)` and `Builder.failFast(...)`.
> - `JWKSRefreshException` is renamed to `JWKSFetchException`. Discovery failures throw `OpenIDConnectException`; JWKS-hop failures throw `JWKSFetchException`. Neither type wraps the other. `JWKS.refresh()` can therefore throw either type depending on which phase failed.
> - `JWKS.fromIssuer` / `JWKS.fromWellKnown` perform discovery at most once successfully over the lifetime of the instance, then cache `jwks_uri` and refresh against it directly. Until discovery succeeds, every refresh re-attempts discovery; once it succeeds, the URL is locked.
> - `refreshTimeout` bounds only the JWKS hop. Discovery is bounded by `HttpURLConnection` connect/read timeouts, tunable via the `Consumer<HttpURLConnection>` customizer.
> - **Removed:** `org.lattejava.jwt.oauth2.AuthorizationServerMetaData`, `org.lattejava.jwt.oauth2.ServerMetaDataHelper`, the entire `org.lattejava.jwt.oauth2` package (also removed from `module-info.java`), and `org.lattejava.jwt.jwks.JSONWebKeySetHelper`. Migration paths above.

## 7. Out of scope / future work

- **Automatic re-discovery after a successful first discovery.** If a server rotates `jwks_uri` after the URL has been locked (per §3.4), `JWKS` will not detect this — it will keep refreshing against the cached URL and (eventually) fail. The caller must rebuild the `JWKS`. A future spec may add an opt-in re-discovery trigger (e.g., re-discover on N consecutive `NON_2XX` failures, or on an explicit signal). Not in 7.0.
- **Async / non-blocking discovery.** `OpenIDConnect.discover(...)` is synchronous and blocking. A future overload returning `CompletableFuture<OpenIDConnectConfiguration>` may be added if a use case emerges.
- **Streaming fetch hooks.** `JWKS.fetchOnce` returns a fully-materialized list. There is no streaming variant.
- **`OpenIDConnectConfiguration.discover()` instance method.** Discovery is a static operation that *returns* the configuration; chaining a method off the type that does not yet exist is an awkward shape. The static entry points on `OpenIDConnect` are the supported front door.
- **Promoting `otherClaims` keys to typed accessors.** §1.1 deliberately ships a typed subset. Fields routinely consumed by callers can be promoted to typed accessors in a 7.x release as demand surfaces; the change is non-breaking.
- **Migrate to `java.net.http.HttpClient`.** The library uses `HttpURLConnection` for both discovery and JWKS hops, and the customizer hook is typed against it. Switching to the modern client changes timeout, redirect, proxy, and exception semantics, and is large enough to warrant its own spec. Deferred from 7.0.
