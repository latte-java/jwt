# JWT Library Architecture Redesign

**Date:** 2026-04-10
**Updated:** 2026-04-21 (security/usability review pass 2; pre-implementation review pass -- RFC 8725 section-number corrections, `VerifierResolver` collapse of decode overloads, `Clock` builder option replaces `TimeMachineJWTDecoder`, `fromMap()` shape validation table, case-sensitivity rule, `crit` structural validation, `Signer`/`Verifier` javadoc contracts, thumbprint input-canonicalization note)
**Version:** 7.0.0 (major breaking change)
**Scope:** Full library -- JWT core, JWK, OAuth2 metadata, PEM/DER enhancements. JWE (JSON Web Encryption, RFC 7516) is out of scope for 7.0.
**License:** All new files use the MIT license.

## Design Goals

1. **Zero required dependencies.** Ship a built-in JSON parser. No Jackson, no Gson, nothing.
2. **Bring your own JSON.** A `JSONProcessor` strategy interface lets users plug in Jackson/Gson/etc.
3. **Spec-aligned types.** Time claims (`exp`, `nbf`, `iat`) use `Instant` instead of `ZonedDateTime`. No custom serializers needed.
4. **Performance.** The Map-based serialization path avoids reflection overhead. The built-in parser targets competitive performance with Jackson for small payloads (JWT-sized).
5. **Extensibility.** Algorithm as an interface (not enum). Strategy pattern for JSON. Clean builder APIs.
6. **Immutable models.** JWT and Header are immutable, built via builders. Built instances are thread-safe by construction. Builders are reusable -- calling `build()` produces an independent instance, and the builder remains valid for further modification and rebuilding -- but builders themselves are not thread-safe and should not be shared across threads.
7. **No private JDK classes.** Eliminate all use of `sun.*`, `com.sun.*`, and `jdk.internal.*` in production code. X.509 certificate parsing and generation uses our own DER encoder/decoder.

## Breaking Changes from 6.x

This is a major version bump. The following changes are intentionally breaking:

| Change | 6.x | 7.0 | Impact |
|--------|-----|-----|--------|
| `Algorithm` | Enum | Interface with constants | `switch` on Algorithm no longer compiles; use `algorithm.name()` or `.equals()` |
| `Algorithm.getName()` | Returns JCA string (e.g., `"SHA256withRSA"`) | `Algorithm.name()` returns JWA identifier (e.g., `"RS256"`) | Code using `getName()` for JCA lookups gets wrong values if mechanically migrated. JCA strings are now internal to Signer/Verifier implementations. |
| `KeyType` | Enum | Interface with constants | Same as Algorithm |
| `Signer.sign()` | `byte[] sign(String payload)` | `byte[] sign(byte[] message)` | All custom `Signer` implementations must update. The encoder now calls `getBytes(UTF_8)` before passing to `sign()`. |
| `Signer.getAlgorithm()` | Returns `Algorithm` enum | `algorithm()` returns `Algorithm` interface | Method rename + return type change |
| `Signer.getKid()` | Throws `UnsupportedOperationException` by default | `kid()` returns `null` by default | Behavior change: callers no longer need to catch exceptions |
| `JWT` | Mutable POJO with setters, `ZonedDateTime` fields | Immutable, builder pattern, `Instant` fields, long English names (`subject` not `getSubject`, fluent no-prefix style) | Full rewrite of construction and access patterns |
| `Header` | Mutable POJO with `set()` | Immutable, builder pattern, explicit `kid` field | Full rewrite |
| `JWT.getEncoder()` / `JWT.getDecoder()` | Static factories | Removed | Use `new JWTEncoder()` / `new JWTDecoder()` directly |
| Jackson dependency | Required (compile) | Removed | Users who need Jackson implement `JSONProcessor` |
| `Buildable<T>` interface | Used by `JSONWebKey`, `PEM` | Removed | Replaced by `Builder` inner classes |

## 1. Algorithm (Interface)

Replaces the current `Algorithm` enum. Extensible for custom algorithms.

```java
public interface Algorithm {
    /**
     * The JWA (JSON Web Algorithm) identifier as registered in the
     * IANA "JSON Web Signature and Encryption Algorithms" registry.
     * This is the value used in the JWT header "alg" parameter.
     *
     * <p>Examples: {@code "RS256"}, {@code "ES384"}, {@code "Ed25519"}</p>
     *
     * <p>This is <em>not</em> the JCA (Java Cryptography Architecture) algorithm
     * string (e.g., {@code "SHA256withRSA"}). JCA strings are internal to each
     * Signer/Verifier implementation and are not exposed by this interface.</p>
     *
     * @return the JWA algorithm name
     */
    String name();

    // Standard constants
    Algorithm HS256 = new StandardAlgorithm("HS256");
    Algorithm HS384 = new StandardAlgorithm("HS384");
    Algorithm HS512 = new StandardAlgorithm("HS512");
    Algorithm RS256 = new StandardAlgorithm("RS256");
    Algorithm RS384 = new StandardAlgorithm("RS384");
    Algorithm RS512 = new StandardAlgorithm("RS512");
    Algorithm PS256 = new StandardAlgorithm("PS256");
    Algorithm PS384 = new StandardAlgorithm("PS384");
    Algorithm PS512 = new StandardAlgorithm("PS512");
    Algorithm ES256 = new StandardAlgorithm("ES256");
    Algorithm ES384 = new StandardAlgorithm("ES384");
    Algorithm ES512 = new StandardAlgorithm("ES512");
    Algorithm Ed25519 = new StandardAlgorithm("Ed25519");
    Algorithm Ed448  = new StandardAlgorithm("Ed448");
    Algorithm ES256K = new StandardAlgorithm("ES256K");

    /**
     * Look up by JWA name. Returns the pre-built standard constant if the name matches
     * one of the 14 standard algorithms (enabling {@code ==} comparison for standard
     * algorithms). Returns a new instance for unrecognized names.
     */
    static Algorithm of(String name) { ... }
}
```

`StandardAlgorithm` is a package-private class with `equals`/`hashCode` based on `name()`.

The `Algorithm` interface is purely the JWA identifier for the header -- it carries no crypto configuration. Each Signer/Verifier implementation internally maps the JWA name to the appropriate JCA algorithm string (e.g., `RS256` → `"SHA256withRSA"`, `HS256` → `"HmacSHA256"`). This separation is intentional: the JCA string is a Java platform implementation detail, while the JWA name is the wire-format identifier that appears in JWT headers. Users interact with JWA names; the JCA mapping is an internal concern.

Custom algorithms: `Algorithm.of("MY_ALG")` or implement the interface directly.

### Algorithm Identity

`Algorithm.of(name)` returns the pre-built constant for standard algorithms, so `==` works:

```java
Algorithm.of("RS256") == Algorithm.RS256  // true -- same reference
Algorithm.of("MY_ALG") == Algorithm.of("MY_ALG")  // false -- different instances
Algorithm.of("MY_ALG").equals(Algorithm.of("MY_ALG"))  // true -- equals uses name()
```

For non-standard algorithms, use `.equals()`. Since `Algorithm` is no longer an enum, `switch` statements require `algorithm.name()`:

```java
switch (algorithm.name()) {
    case "RS256": ...
    case "ES256": ...
}
```

### Case Sensitivity

`Algorithm.of()` performs an exact-case lookup against the standard constants. `Algorithm.of("rs256")` produces a non-standard instance with `name() == "rs256"` -- it does **not** match `Algorithm.RS256`. No built-in verifier's `canVerify()` returns true for mis-cased variants, so tokens with lowercase/mixed-case `alg` values are rejected with `MissingVerifierException`. This matches RFC 7515 §4.1.1's case-sensitive treatment of registered header parameter values.

The same rule applies to `KeyType.of()`.

### Security: the "none" algorithm

No special `Algorithm.NONE` constant is defined. `Algorithm.of("none")` creates a normal instance. This is safe by design: the decoder always requires a verifier where `canVerify(algorithm)` returns true. Since no built-in verifier handles "none", the decoder throws `MissingVerifierException`. An attacker cannot exploit `alg: none` unless the application explicitly creates a verifier for it.

For decoding unsigned JWTs (e.g., inspecting claims without verification), see `JWTDecoder.decodeUnsecured()` in [Section 5](#5-encoder--decoder).

### EdDSA: Fully-Specified Algorithm Identifiers (RFC 9864)

Per [RFC 9864](https://www.rfc-editor.org/rfc/rfc9864.html) (October 2025), the polymorphic `"EdDSA"` algorithm identifier is **deprecated** in the IANA JOSE registry. The fully-specified replacements are `"Ed25519"` and `"Ed448"`, which this library uses as the standard constants.

No `Algorithm.EdDSA` constant is defined. If a token arrives with `"alg": "EdDSA"` in the header, `Algorithm.of("EdDSA")` creates a non-standard instance. No built-in verifier's `canVerify()` will match it, and the decoder throws `MissingVerifierException`. Users who need to handle legacy `"EdDSA"` tokens can build a custom verifier that accepts it -- that is an application-level decision, not a library concern.

The signers always emit the fully-specified identifier (`"Ed25519"` or `"Ed448"`) in the JWT header.

**Handling legacy `"EdDSA"` tokens:** The library architecture supports this without any built-in changes. A user who receives tokens with `"alg": "EdDSA"` can wrap their Ed25519 verifier to accept the legacy identifier:

```java
Verifier legacyEdDSA = new Verifier() {
    private final EdDSAVerifier delegate = new EdDSAVerifier(publicKey);

    public boolean canVerify(Algorithm algorithm) {
        return "EdDSA".equals(algorithm.name()) || delegate.canVerify(algorithm);
    }

    public void verify(Algorithm algorithm, byte[] message, byte[] signature) {
        delegate.verify(delegate.algorithm(), message, signature);
    }
};
```

This is an application-level decision because the old `"EdDSA"` identifier is polymorphic -- it could mean Ed25519 or Ed448. The application knows which curve to expect; the library does not.

### Algorithms with Runtime JCE Provider Requirements

Several standard algorithms depend on JCE provider support that may not be present in every JDK:

| Algorithm | Requirement | Failure mode if unsupported |
|-----------|-------------|-----------------------------|
| `Ed25519` | JDK 15+ `SunEC` (JEP 339), or BouncyCastle on JDK 8/11 | `NoSuchAlgorithmException` at sign/verify time |
| `Ed448` | JDK 15+ `SunEC` (JEP 339) — SHAKE256 is used internally by `SunEC`'s Ed448 implementation and does not need separate registration. BouncyCastle required on JDK 8/11. | `NoSuchAlgorithmException` at sign/verify time |
| `ES256K` | BouncyCastle on JDK 15+ (Oracle removed `secp256k1` from `SunEC` in JDK 15 because it is not approved for TLS) | `NoSuchAlgorithmException` or `InvalidAlgorithmParameterException` at sign/verify time |

The library does not probe for provider availability at construction time. Signers and verifiers for these algorithms are constructable on any JDK; the first sign/verify call fails with a clear `NoSuchAlgorithmException` (propagated as `JWTSigningException` / `JWTVerifierException`) if the required provider is missing. This matches the standard JCE failure pattern and keeps the library dependency-free.

**Note on OIDC `at_hash`/`c_hash` for Ed448:** OIDC `at_hash`/`c_hash` for Ed448 requires SHAKE256, which stock `SunEC` does *not* register as a public `MessageDigest` service (even on JDK 15+, where Ed448 sign/verify works out of the box because `SunEC` uses its internal package-private SHAKE256). The library ships a bundled FIPS 202 SHAKE256 implementation for zero-dependency operation but **prefers a JCE-registered SHAKE256 provider when one is available** (e.g., BouncyCastle or BC-FIPS) so that FIPS-mode deployments keep `at_hash`/`c_hash` inside the validated cryptographic module. See §10 "Internal SHAKE256 Implementation" for the provider-preference probe, caching, and self-test rules. `OpenIDConnect.generate_hash` calls the internal helper for the Ed448 branch and continues to use `MessageDigest.getInstance` for the SHA-2 branches.

`ES256K` is included as a standard constant because:
- It is registered in the IANA JOSE algorithm registry (RFC 8812).
- It is actively used in decentralized identity (DID), verifiable credentials, and cryptocurrency/web3 contexts.
- Omitting it would force users to implement a redundant `Signer`/`Verifier` that duplicates the existing `ECSigner`/`ECVerifier` logic — the only difference is the curve encoded in the key.
- The failure mode for users without provider support is clean and actionable.

## 2. JWT (Immutable + Builder)

```java
public class JWT {
    // Registered claims -- long English names, spec-aligned types
    private final String issuer;          // "iss"
    private final String subject;         // "sub"
    private final List<String> audience;  // "aud": always stored as List, wire form preserved via audienceWireForm
    private final AudienceWireForm audienceWireForm;  // STRING or ARRAY, determines serialization
    private final Instant expiresAt;      // "exp"
    private final Instant notBefore;      // "nbf"
    private final Instant issuedAt;       // "iat"
    private final String id;              // "jti"
    private final Map<String, Object> customClaims;

    // Header -- populated on decode, not serialized with claims
    private final Header header;

    // --- Fluent getters (no "get" prefix, long English names) ---
    public String issuer() { ... }
    public String subject() { ... }
    public Instant expiresAt() { ... }
    public Instant notBefore() { ... }
    public Instant issuedAt() { ... }
    public String id() { ... }
    public Header header() { ... }

     * audience, returns an empty list (never null).
     *

     * audience is serialized as {@code "aud": "foo"}, a list audience is
     * serialized as {@code "aud": ["foo", "bar"]}.</p>

    public List<String> audience() { ... }

    // These look up by name, checking registered claims first, then customClaims.

    public String getString(String name) { ... }

    public Long getLong(String name) { ... }
    public Float getFloat(String name) { ... }
    public Double getDouble(String name) { ... }

    public List<Object> getList(String name) { ... }

     * element does not match.
     */

    // --- Maps ---
    /** All claims (registered + custom) merged, with raw Java types (Instant, etc.). */
    public Map<String, Object> claims() { ... }

    /** All claims with JSON-safe values (Instant -> epoch seconds). Ready for serialization. */
    public Map<String, Object> toSerializableMap() { ... }

    // --- Convenience ---
    public boolean isExpired() { ... }
    public boolean isExpired(Instant now) { ... }
    public boolean isUnavailableForProcessing() { ... }
    public boolean isUnavailableForProcessing(Instant now) { ... }

    // --- Factory ---
    /** Hydrate a JWT from a deserialized map (used by decoder). */
    public static JWT fromMap(Map<String, Object> map, Header header) { ... }

    // --- equals / hashCode / toString ---
    // equals/hashCode are based on claim fields ONLY. The Header is excluded:
    // a JWT is a claim set, not a signed token. A JWT constructed via the
    // Builder (which has no Header) is equal to a decoded JWT (which has a
    // populated Header) when their claim fields match. Callers who need to
    // compare signing metadata should compare jwt.header() explicitly.
    // toString() uses the built-in LatteJSONProcessor for JSON pretty-print.
    // This output is for debugging/logging; use JWTEncoder for wire-format serialization.
    // toString() does not depend on JWTConfig or any user-provided JSONProcessor.

    public static Builder builder() { return new Builder(); }

    public static class Builder {
        public Builder issuer(String issuer) { ... }

        /**
         * Add a claim. If the name matches a registered claim (iss, sub, aud, exp,
         * nbf, iat, jti), the value is routed to the corresponding typed field with
         * type coercion (see table below). Throws IllegalArgumentException if the
         * value cannot be coerced to the registered claim's type. Unrecognized names
         * are stored in customClaims. A registered claim name can never collide with
         * customClaims.

        public JWT build() { ... }

```

### Builder.claim() Coercion Rules

When `claim(name, value)` is called with a registered claim name, the value is coerced to the target type:

| Claim name | Target type | Accepted value types | Coercion |
|------------|------------|---------------------|----------|
| `iss`, `sub`, `jti` | `String` | `String` | Direct assignment |
| `iss`, `sub`, `jti` | `String` | Any other type | `IllegalArgumentException` |
| `exp`, `nbf`, `iat` | `Instant` | `Instant` | Direct assignment |
| `exp`, `nbf`, `iat` | `Instant` | `Number` (Long, Integer, BigInteger, BigDecimal, Double) | `Instant.ofEpochSecond(value.longValue())` |
| `exp`, `nbf`, `iat` | `Instant` | `ZonedDateTime` | `value.toInstant()` (eases migration from 6.x) |
| `exp`, `nbf`, `iat` | `Instant` | Any other type | `IllegalArgumentException` |
| `aud` | `List<String>` | `String` | Stored as single-element `List`, wire form recorded as STRING |
| `aud` | `List<String>` | `List<String>` | Stored as `List`, wire form recorded as ARRAY |
| `aud` | `List<String>` | Any other type | `IllegalArgumentException` |

Unrecognized claim names are stored as-is in `customClaims` with no type coercion.

```java
// These are equivalent:
JWT.builder().expiresAt(Instant.ofEpochSecond(1700000000)).build();
JWT.builder().claim("exp", 1700000000L).build();
JWT.builder().claim("exp", Instant.ofEpochSecond(1700000000)).build();
```

### Audience Handling

The `aud` claim is always exposed to callers as `List<String>`, but the original wire form (JSON string vs. JSON array) is preserved for round-trip fidelity:

- `builder.audience("single")` -> stored as `["single"]`, wire form STRING -> serialized as `"aud": "single"`
- `builder.audience(List.of("single"))` -> stored as `["single"]`, wire form ARRAY -> serialized as `"aud": ["single"]`
- `builder.audience(List.of("a", "b"))` -> stored as `["a", "b"]`, wire form ARRAY -> serialized as `"aud": ["a", "b"]`

On deserialization, `fromMap()` records the wire form: a JSON string sets wire form to STRING (and stores a one-element list), a JSON array sets wire form to ARRAY. `toSerializableMap()` emits whichever form the JWT was built/decoded with.

RFC 7519 §4.1.3 permits either form on the wire, and the audience list does not require unique values (the RFC does not mandate uniqueness). We preserve duplicates and order.

Rationale for always returning `List<String>`: matches auth0 java-jwt and Nimbus, eliminates the caller-side "is it a string or a list?" branching, and makes the common case (checking for membership) a one-liner: `jwt.audience().contains("my-service")`.

The map returned by `claims()` always carries `aud` as a `List<String>` (consistent with the `audience()` accessor) whenever present. The STRING-vs-ARRAY wire form is only consulted by `toSerializableMap()` to pick the serialized shape -- it is not surfaced through `claims()`.

### fromMap() Shape Validation

`JWT.fromMap()` validates the JSON shape of every registered claim. Mismatches throw `InvalidJWTException` -- the library does not attempt silent coercion (matches the behavior of jjwt, nimbus-jose-jwt, and auth0 java-jwt, all of which reject wrong-shape registered claims):

| Claim | Required JSON shape | Notes |
|-------|--------------------|-------|
| `iss`, `sub`, `jti` | string | Non-string throws `InvalidJWTException` |
| `exp`, `nbf`, `iat` | number (any `Number` subtype: Long, Integer, BigInteger, BigDecimal, Double) | Converted to `Instant` via `longValue()`. Non-numeric throws `InvalidJWTException` |
| `aud` | string OR array of strings | Any other shape (object, number, boolean, mixed-type array) throws `InvalidJWTException` |
| Custom / unknown | any | Passed through as-is from the JSON processor into `customClaims` |

`Header.fromMap()` applies the same rule to header parameters: `alg`, `typ`, `kid`, `cty`, `x5t`, `x5t#S256`, `x5u` must be strings if present; `x5c` must be an array of strings; `crit` must be an array of strings (see §3 "Critical Header Parameter" for full `crit` structural rules). Custom parameters pass through unchanged.

### NumericDate Fractional Seconds

RFC 7519 §2 defines NumericDate as "a JSON numeric value representing the number of seconds from 1970-01-01T00:00:00Z UTC until the specified UTC date/time, ignoring leap seconds," and explicitly permits non-integer (fractional) values.

This library **truncates fractional seconds** when converting to `Instant`:

- `fromMap()` calls `Number.longValue()` on time-claim values, discarding the fractional component.
- `toSerializableMap()` writes `Instant.getEpochSecond()` (integer seconds) on output.

A token with `"exp": 1700000000.5` is parsed as `Instant.ofEpochSecond(1700000000)` and re-serialized as `"exp": 1700000000`. Fractional precision is lost silently; this is intentional and matches the behavior of jjwt, auth0 java-jwt, and nimbus-jose-jwt (all of which use `java.util.Date` and truncate similarly).

Tokens with fractional `exp`/`nbf`/`iat` values are essentially never seen in the wild. Callers who require sub-second precision can access the raw value via `getBigDecimal("exp")` before the typed getter is invoked.

### Immutability of Returned Collections

All map and list accessors return unmodifiable views. Callers cannot mutate the JWT's internal state through returned references:

- `claims()` returns `Collections.unmodifiableMap(...)` (shallow copy)
- `toSerializableMap()` returns an unmodifiable map (new map, not a view of internals)
- `audience()` returns `Collections.unmodifiableList(...)`
- `Header.parameters()` and `Header.toSerializableMap()` same pattern
- `JSONWebKey.parameters()`, `JSONWebKey.x5c()` same pattern

This is a shallow guarantee -- map values that are mutable objects (e.g., nested `Map` or `List` from custom claims) are not defensively copied. For JWT-sized payloads with standard JSON types this is sufficient.

### toSerializableMap() Behavior

Registered claims are written first (using `LinkedHashMap` to preserve insertion order), then custom claims. Since `Builder.claim()` routes registered names to their typed fields (never to `customClaims`), no collision between registered and custom claims is possible in `toSerializableMap()`. Null-valued claims are omitted.

### Key changes from current

| Aspect | Current (6.x) | New (7.0) |
|--------|---------------|-----------|
| Fields / accessors | `subject`, `issuer`, `expiration` (with `get`/`set` prefix) | `subject()`, `issuer()`, `expiresAt()` (fluent, no prefix) |
| Time type | `ZonedDateTime` | `Instant` (+ `long` builder overloads) |
| Mutability | Mutable POJO with setters | Immutable, builder pattern (reusable builders) |
| Annotations | `@JsonProperty`, `@JsonSerialize`, etc. | None |
| Serialization | Jackson ObjectMapper directly | `toSerializableMap()` -> `JSONProcessor` |
| Custom claims map | `otherClaims` | `customClaims` (not directly exposed) |
| Audience | `Object` with no typed accessors | `List<String>` via `audience()`, wire form preserved |
| Static factories | `JWT.getEncoder()`, `JWT.getDecoder()` | Removed -- construct `new JWTEncoder()` / `new JWTDecoder()` directly |
| `toString()` | Uses `Mapper.prettyPrint()` (Jackson) | Uses built-in `LatteJSONProcessor` always |

## 3. Header (Immutable + Builder)

```java
public class Header {
    private final Algorithm alg;
    private final String typ;       // default "JWT"
    private final String kid;
    private final Map<String, Object> customParameters;

    // Fluent getters
    public Algorithm alg() { ... }
    public String typ() { ... }
    public String kid() { ... }

    // Custom parameter access
    public Object get(String name) { ... }
    public String getString(String name) { ... }

    // Maps
    /** All parameters (alg, typ, kid, custom) merged. */
    public Map<String, Object> parameters() { ... }
    public Map<String, Object> toSerializableMap() { ... }

    // Factory
    public static Header fromMap(Map<String, Object> map) { ... }

    // equals / hashCode / toString (toString uses built-in LatteJSONProcessor)

    public static Builder builder() { ... }

    public static class Builder {
        public Builder alg(Algorithm algorithm) { ... }
        public Builder typ(String type) { ... }
        public Builder kid(String keyId) { ... }
        public Builder parameter(String name, Object value) { ... }

```

Header uses RFC 7515 terminology: "parameters" (not "claims"). Custom parameter storage is `customParameters` internally, accessed via `get(name)` and `parameters()` (which merges all parameters into a single map).

### Header.fromMap() Behavior

`Header.fromMap()` converts the `"alg"` string value to an `Algorithm` instance via `Algorithm.of()`. If `"alg"` is missing from the map, `fromMap()` throws `InvalidJWTException` -- the `alg` parameter is REQUIRED per RFC 7515 Section 4.1.1.

### Critical Header Parameter (`crit`) — RFC 7515 Section 4.1.11

Per RFC 7515, if a JWT header contains a `"crit"` array, the recipient MUST understand and process every header parameter name listed, or reject the JWS. Failing to enforce this is an active CVE category (CVE-2026-32597, CVE-2026-35042, GHSA-9ggr-2464-2j32) -- libraries that silently accept tokens with unrecognized `crit` entries enable split-brain verification in heterogeneous systems.

**Behavior:** The decoder checks for `"crit"` in the parsed header. If present and non-empty, every name in the array must appear in the decoder's set of understood critical parameters. If any name is unrecognized, the decoder throws `InvalidJWTException`. Since this library does not natively process any extension header parameters, the default set is empty -- meaning any token with a non-empty `crit` array is rejected unless the caller explicitly opts in.

**Configuration:** `JWTDecoder` accepts an optional `Set<String>` of understood critical header parameter names:

```java
// Default -- rejects any token with crit (safe default)
JWTDecoder decoder = new JWTDecoder();

// Opt in to specific critical headers
JWTDecoder decoder = new JWTDecoder.Builder()
    .criticalHeaders(Set.of("b64", "http://openbanking.org.uk/iat"))
    .build();
```

The decoder validates that the listed names are present; the application is responsible for actually processing the extension semantics. This follows the same pattern as JJWT's `critical().add(...)` API.

**Structural validation.** Beyond the understood-parameters check, the decoder validates the structure of the `crit` value itself. All of the following throw `InvalidJWTException`:

- `crit` is present but is not a JSON array
- Any element of `crit` is not a JSON string
- `crit` contains a duplicate entry
- `crit` contains the name of any registered RFC 7515 header parameter (`alg`, `jku`, `jwk`, `kid`, `x5u`, `x5c`, `x5t`, `x5t#S256`, `typ`, `cty`, `crit`) -- RFC 7515 §4.1.11 prohibits this explicitly
- `crit` is an empty array -- RFC 7515 §4.1.11 forbids empty `crit` arrays. (jjwt and nimbus silently accept this; we reject because an empty `crit` indicates an incorrectly constructed token.)

## 4. JSONProcessor (Strategy Interface)

```java
public interface JSONProcessor {
    /**
     * Serialize a map to UTF-8 JSON bytes.
     *
     * @throws JSONProcessingException on serialization failure
     */
    byte[] serialize(Map<String, Object> object) throws JSONProcessingException;

    /**
     * Deserialize UTF-8 JSON bytes into a Map. The top-level JSON value MUST
     * be an object; top-level arrays, strings, numbers, booleans, or null MUST
     * cause JSONProcessingException. JWT payloads and headers are always JSON
     * objects per RFC 7519 §7.2, so this constraint imposes no real-world
     * limitation on the decoder.
     *
     * @throws JSONProcessingException on malformed JSON or non-object top-level value
     */
    Map<String, Object> deserialize(byte[] json) throws JSONProcessingException;
}
```

`JSONProcessingException` extends `JWTException` (unchecked). The `throws` clause is documentary -- it tells implementors what to throw. The encoder/decoder catch and propagate these directly.

Implementations MUST be stateless and thread-safe: the encoder/decoder may invoke `serialize()` / `deserialize()` concurrently on a single `JSONProcessor` instance.

### Configuration

Two configuration tiers, no global mutable state:

1. **Constructor injection**: `new JWTEncoder(myProcessor)` — caller supplies a `JSONProcessor`.
2. **Built-in default**: `new JWTEncoder()` — uses `LatteJSONProcessor`, always available, zero dependencies.

```java
// Simplest usage -- built-in parser, zero config
JWTEncoder encoder = new JWTEncoder();
JWTDecoder decoder = new JWTDecoder();

// Explicit override per instance
JWTEncoder encoder = new JWTEncoder(new JacksonJSONProcessor());
```

**No global config.** Earlier drafts proposed a `JWTConfig.setDefaultJSONProcessor()` global. It was removed because:
- Constructor injection already handles every legitimate use case.
- Global mutable state introduces race conditions on startup/reconfiguration and "was this set before my code ran?" debugging.
- Applications with DI containers (Spring, Guice, Dagger) prefer to wire encoders/decoders themselves; a library global competes with that.
- Users who want Jackson everywhere can build `JWTEncoder` / `JWTDecoder` once and share them (they are thread-safe per §5).

**No SPI / ServiceLoader.** Explicit construction covers all use cases without the debugging pain and classpath ambiguity that ServiceLoader introduces. JJWT uses ServiceLoader and has documented issues with fat JARs and runtime performance. We avoid this.

### Internal Canonical JSON Writer

The `JSONProcessor` interface is intentionally permissive -- implementations may produce any valid JSON for a given input. For computations that require **deterministic, canonical** JSON output (notably RFC 7638 JWK thumbprints), the library uses a separate internal `CanonicalJSONWriter` that is not pluggable:

- Lexicographically sorted object keys
- No whitespace
- UTF-8 byte output
- Only the primitive types needed for RFC 7638 (strings, numbers, booleans, null)

This writer is ~40 LOC, lives under `org.lattejava.jwt.internal`, and is never exposed via public API. It exists to prevent thumbprint computation from being silently broken by a user-supplied `JSONProcessor` that reorders keys, adds whitespace, or otherwise alters serialization. See [Section 10](#10-jwtutils) for details.

### Built-in Implementation

`LatteJSONProcessor` -- a hand-rolled JSON reader + writer.

**Writer:** Iterates map entries, writes JSON with proper string escaping, handles nested maps/lists recursively. No intermediate allocations beyond the output buffer.

**Reader:** Recursive descent parser. Dispatches on first character: `{` (object), `[` (array), `"` (string), digit/`-` (number), `t`/`f` (boolean), `n` (null).

**Number handling:** Always `BigInteger` for integer values, always `BigDecimal` for decimal values. This matches the current Jackson configuration (`USE_BIG_INTEGER_FOR_INTS` + `USE_BIG_DECIMAL_FOR_FLOATS`) and ensures no precision loss for custom claims. The typed accessors on JWT (`getLong()`, `getInteger()`) handle conversion from BigInteger.

**JSON type mapping:**

| JSON | Java |
|------|------|
| object | `LinkedHashMap<String, Object>` |
| array | `ArrayList<Object>` |
| string | `String` |
| integer number | `BigInteger` |
| decimal number | `BigDecimal` |
| boolean | `Boolean` |
| null | `null` |

The implementation is expected to be small and efficient. Since the JSON layer is not the bottleneck (crypto dominates, see §13), the priority is correctness, strict input handling (depth limit, duplicate-key rejection), and stability over raw throughput.

### User-provided Example (Jackson)

```java
public class JacksonJSONProcessor implements JSONProcessor {
    private final ObjectMapper mapper = new ObjectMapper();

    @Override
    public byte[] serialize(Map<String, Object> object) {
        try {
            return mapper.writeValueAsBytes(object);
        } catch (JsonProcessingException e) {
            throw new JSONProcessingException("JSON serialization failed", e);
        }
    }

    @Override
    public Map<String, Object> deserialize(byte[] json) {
        try {
            return mapper.readValue(json, new TypeReference<>() {});
        } catch (IOException e) {
            throw new JSONProcessingException("JSON deserialization failed", e);
        }
    }
}
```

### VerifierResolver

Decouples verifier *selection* from the decoder. Replaces the three separate decode overloads (single `Verifier`, `Map<String, Verifier>`, `Function<Header, Verifier>`) that existed in earlier drafts of this spec.

```java
/**
 * Resolves a Verifier for a given JWT header. Returning null means "no verifier
 * for this header" -- the decoder then throws MissingVerifierException.
 */
public interface VerifierResolver {
    /** Return the verifier for this header, or null if none applies. */
    Verifier resolve(Header header);

    /** Always return the given verifier (after canVerify check). */
    static VerifierResolver of(Verifier v) {
        return header -> v.canVerify(header.alg()) ? v : null;
    }

    /** Look up a verifier by the header's kid. Returns null if kid is missing
     *  or not present in the map. */
    static VerifierResolver byKid(Map<String, Verifier> map) {
        return header -> header.kid() == null ? null : map.get(header.kid());
    }

    /** Arbitrary resolver, for cases where kid-based lookup is insufficient
     *  (e.g., resolution by iss, by x5t, or via an out-of-band registry). */
    static VerifierResolver from(Function<Header, Verifier> fn) {
        return fn::apply;
    }
}
```

Usage:

```java
// Single verifier
decoder.decode(token, VerifierResolver.of(verifier));

// Kid-keyed map (common for JWKS-backed verification)
decoder.decode(token, VerifierResolver.byKid(verifiersByKid));

// Arbitrary resolution
decoder.decode(token, VerifierResolver.from(header ->
    myRegistry.lookupByIssuer(header.kid(), header.alg())));

// With post-decode validator
decoder.decode(token, VerifierResolver.of(verifier), jwt -> {
    if (!"expected-issuer".equals(jwt.issuer()))
        throw new InvalidJWTException("Unexpected issuer");
});
```

**Breaking change from 6.x:** the `decoder.decode(token, verifier1, verifier2, ...)` varargs form is gone. Callers compose via `Verifiers.anyOf(...)` (see §6) wrapped in `VerifierResolver.of(...)`, or use `VerifierResolver.byKid(...)` / `VerifierResolver.from(...)` for kid-keyed or custom resolution.

`JWTDecoder` and `JWTEncoder` are lightweight objects intended to be instantiated per-use. All fields are `final`. The decoder is not designed for shared/long-lived use, but its immutability means there is no thread-safety hazard if it is shared.

For advanced configuration, use the builder:

```java
JWTDecoder decoder = new JWTDecoder.Builder()
    .jsonProcessor(myProcessor)
    .clock(Clock.fixed(Instant.parse("2020-01-01T00:00:00Z"), ZoneOffset.UTC))
    .clockSkew(Duration.ofSeconds(30))
    .criticalHeaders(Set.of("b64"))
    .expectedType("at+jwt")
    .expectedAlgorithms(Set.of(Algorithm.RS256, Algorithm.ES256))
    .maxTokenBytes(131072)
    .maxJsonDepth(24)
    .allowDuplicateJsonKeys(false)
    .build();
```

The builder exposes a setter per field; `build()` returns an immutable `JWTDecoder`. The constructors remain for the common cases; the builder adds access to the full configuration surface.

The `clock(Clock)` option replaces the 6.x `TimeMachineJWTDecoder` subclass. Default is `Clock.systemUTC()`. **This option exists only for tests and time travelers** -- production code should leave the default in place. Using a fixed clock lets tests exercise time-dependent validation (`exp`, `nbf`, clock-skew interactions) deterministically, and it composes cleanly with every other builder option (which the old subclass approach did not).

### Decoder Configuration Defaults

| Field | Default | Rationale |
|-------|---------|-----------|
| `jsonProcessor` | `LatteJSONProcessor` | Zero-dependency default |
| `clock` | `Clock.systemUTC()` | Production default. Test/time-travel uses `Clock.fixed(...)`. |
| `clockSkew` | `Duration.ZERO` | Strict by default; matches jjwt and auth0 java-jwt. Nimbus defaults to 60s — no industry consensus. Callers enable skew explicitly. Applied symmetrically to `exp` and `nbf` when > 0. |
| `criticalHeaders` | empty set | RFC 7515 §4.1.11 requires rejecting unknown `crit` entries; empty set means any non-empty `crit` array is rejected unless opted in. |
| `expectedType` | `null` | No `typ` enforcement by default. When set, header `typ` must equal this value (case-insensitive per RFC 7515 §4.1.9). |
| `expectedAlgorithms` | `null` | No algorithm whitelist by default. When set, header `alg` must be in this set; rejection occurs before verifier selection. Defense-in-depth per RFC 8725 §3.1. |
| `maxTokenBytes` | `262144` (256 KiB) | Generous enough for any realistic JWT (including OIDC tokens with embedded claims); rejects pathological inputs before base64 decoding. |
| `maxJsonDepth` | `16` | Real-world JWTs rarely exceed depth 6; 16 gives 2× headroom. Prevents stack-exhaustion DoS from deeply nested custom claims. |
| `allowDuplicateJsonKeys` | `false` | RFC 7519 §5.2 recommends rejecting; duplicate-key smuggling is a known attack vector. Opt-in `true` exists for wire-compatibility emergencies. |

No maximum is imposed on `clockSkew` -- callers are responsible for choosing reasonable values. RFC 8725 does not specify a clock-skew recommendation; common practice is "a few seconds to a minute" for real deployments.

### HeaderCustomizer (Encoder Consumer)

```java
/**
 * Subset of Header.Builder exposed to JWTEncoder.encode() callers. Intentionally
 * omits alg() -- the algorithm is always determined by the Signer and cannot be
 * overridden by the caller, preventing accidental mismatch between the header's
 * declared algorithm and the actual signing algorithm.
 *
 * <p>kid() can be called to override or clear (pass null) the kid inherited from
 * the signer. Other parameters (typ, cty, x5t, etc.) can be set freely.</p>
 */
public interface HeaderCustomizer {
    HeaderCustomizer typ(String type);
    HeaderCustomizer kid(String keyId);
    HeaderCustomizer parameter(String name, Object value);
}
```

The encoder constructs an internal `Header.Builder`, pre-populates `alg` from `signer.algorithm()` and `kid` from `signer.kid()`, passes a `HeaderCustomizer` view to the user's consumer, then completes the build. The type system enforces that `alg` is encoder-controlled -- there is no call available on `HeaderCustomizer` that could alter it. Callers who need a non-signer-driven `alg` in the header must construct a custom `Signer` whose `algorithm()` returns the desired value (see the RFC 9864 legacy EdDSA example in [Section 1](#1-algorithm-interface)).

### Base64URL Strictness

All three base64url segments in the encoded JWT are decoded in strict mode:

- **Alphabet:** only `A-Z a-z 0-9 - _` permitted. Any other character (including `+`, `/`, `=`) throws `InvalidJWTException`.
- **No padding:** trailing `=` characters are rejected. RFC 7515 §2 requires "no trailing '=' characters."
- **No whitespace:** spaces, tabs, newlines, or any other whitespace inside any segment throws `InvalidJWTException`.

Lenient base64url decoding is a known source of token ambiguity (two encodings producing the same decoded value) and has enabled smuggling attacks in other libraries. Strict decoding eliminates this class of issue and matches real-world issuer behavior — tokens surviving HTTP header framing cannot contain whitespace or non-URL-safe characters.

### Input Size and Depth Enforcement

- **Token length:** before any parsing, the encoded JWT length is checked against `maxTokenBytes`. Exceeds → `InvalidJWTException` (not `ResponseTooLargeException`, which is reserved for HTTP response size enforcement in `JSONWebKeySetHelper`).
- **JSON depth:** `LatteJSONProcessor` tracks recursion depth during parsing; exceeds `maxJsonDepth` → `JSONProcessingException`. User-supplied `JSONProcessor` implementations are expected to impose their own limits; the decoder does not double-check.
- **Duplicate JSON keys:** `LatteJSONProcessor` rejects duplicates by default. User-supplied processors are expected to honor the same contract when `allowDuplicateJsonKeys` is `false`; the decoder cannot enforce this for external processors.

### Unsecured JWT Decoding

Moved from `JWTUtils.decodePayload()` and `JWT.decodeUnsecured()` to `JWTDecoder.decodeUnsecured()`. This keeps the `JWT` class as a pure model/POJO with no infrastructure dependencies. The method is explicitly named to make the security implications clear. It parses the payload without any signature verification -- for inspection/debugging only.

```java
JWT claims = new JWTDecoder().decodeUnsecured(token);
```

The returned JWT has its `header()` populated, so a separate header-only decode method is unnecessary.

### Encode Flow

1. Build `Header` -- the encoder pre-populates `alg` from `signer.algorithm()` and `kid` from `signer.kid()` (null if the signer has no kid):
   - `encode(jwt, signer)`: uses signer defaults directly
   - `encode(jwt, signer, customizer)`: passes a `HeaderCustomizer` view (no `.alg()` method) to the consumer. The consumer can set `typ`, `kid`, or arbitrary `parameter(name, value)` entries. Because `HeaderCustomizer` has no `.alg()`, the `alg` parameter cannot be modified by the customizer -- the type system enforces that the header `alg` matches the actual signing algorithm.
2. `header.toSerializableMap()` -> `jsonProcessor.serialize()` -> Base64URL encode (no padding, per RFC 7515 Section 2)
3. `jwt.toSerializableMap()` -> `jsonProcessor.serialize()` -> Base64URL encode (no padding)
4. Concatenate `encodedHeader.encodedPayload`
5. Convert the concatenated string to `byte[]` via `getBytes(UTF_8)`, then `signer.sign(bytes)` -> Base64URL encode
6. Return `encodedHeader.encodedPayload.encodedSignature`

### Decode Flow

1. **Size check.** Encoded JWT length > `maxTokenBytes` → `InvalidJWTException`. Runs before any parsing.
2. **Structural check.** Validate each segment is strict base64url (alphabet, no padding, no whitespace).
3. **Split on `.`** -- requires exactly 3 segments. Fewer than 3 (missing signature) throws `MissingSignatureException`. More than 3 (e.g., a JWE compact serialization with 5 parts) throws `InvalidJWTException`.
4. **Decode header.** Base64URL decode header -> `jsonProcessor.deserialize()` -> `Header.fromMap()`. Missing `alg` throws `InvalidJWTException`.
5. **Algorithm whitelist check.** If `expectedAlgorithms` is set and `header.alg()` is not in the set → `InvalidJWTException`. Runs before verifier selection as defense-in-depth.
6. **Type check.** If `expectedType` is set and `header.typ()` does not match (case-insensitive) → `InvalidJWTException`.
7. **`crit` check.** If `crit` is present in the header, validate all listed parameter names against the decoder's `criticalHeaders` set. Unrecognized names throw `InvalidJWTException`.
8. **Resolve verifier.** Call `resolver.resolve(header)`.
   - If the resolver returns null, throw `MissingVerifierException`.
   - If the resolver returns a non-null verifier but `verifier.canVerify(header.alg())` returns false, throw `MissingVerifierException` (defense-in-depth -- catches misbehaving custom resolvers that return a verifier for the wrong algorithm family).
9. **Verify signature** against `headerB64.payloadB64`.
10. **Decode payload.** Base64URL decode payload -> `jsonProcessor.deserialize()` -> `JWT.fromMap()`. `fromMap()` converts `Number` values in `exp`/`nbf`/`iat` to `Instant`.
11. **Time validation.** Validate `exp`/`nbf` with clock skew.
12. **Custom validator.** If a `validator` was provided, call `validator.accept(jwt)` -- the validator throws any `JWTException` subclass to reject the token.
13. **Return `JWT`.**

Signature verification (step 9) runs before payload deserialization (step 10). This ordering is intentional: it ensures that an unauthenticated payload is never parsed into a JWT object that a caller might accidentally trust.

### Post-decode Validation

The optional `Consumer<JWT> validator` parameter runs after signature verification and built-in time validation. The contract: throw any `JWTException` subclass to reject the token. The decoder does not wrap or catch these -- they propagate directly to the caller.

```java
decoder.decode(token, VerifierResolver.of(verifier), jwt -> {
    if (!"expected-issuer".equals(jwt.issuer()))
        throw new InvalidJWTException("Unexpected issuer: " + jwt.issuer());
    if (!jwt.audience().contains("my-service"))
        throw new InvalidJWTException("Token not intended for this service");
});
```

This is optional. The existing single-argument decode methods work without it.

## 6. Signer / Verifier

```java
public interface Signer {
    /**
     * Returns the JWA algorithm for this signer.
     *
     * @return the algorithm
     */
    Algorithm algorithm();

    /**
     * Sign the provided message and return the signature.
     *
     * @param message The message bytes to sign (header.payload encoded as UTF-8).
     * @return The signature bytes.
     * @throws InvalidKeyLengthException if the signer's key is too short for its algorithm
     * @throws InvalidKeyTypeException if the signer's key is not compatible with its algorithm
     * @throws JWTSigningException if an underlying JCE operation fails (e.g., NoSuchAlgorithmException
     *         from a missing JCE provider for Ed25519 / Ed448 / ES256K on an unsupported JDK)
     */
    byte[] sign(byte[] message);

    /**
     * Returns the key ID for this signer, or null if no key ID is set.
     * The encoder uses this to populate the "kid" header parameter.
     *
     * @return the kid, or null
     */
    default String kid() {
        return null;
    }
}

public interface Verifier {
    /**
     * Does this verifier handle the given algorithm? Answers the algorithm
     * question only -- it does NOT validate a specific signature.
     */
    boolean canVerify(Algorithm algorithm);

    /**
     * Verify the signature. Returns normally on success.
     *
     * <p>The {@code algorithm} argument MUST be one for which
     * {@link #canVerify(Algorithm)} returned true; callers (including
     * {@code JWTDecoder}) guarantee this. Implementations that handle only a
     * single algorithm may ignore the argument. Implementations that handle
     * multiple algorithms in the same family (e.g., one RSAVerifier that
     * dispatches across RS256/RS384/RS512) use it as a dispatch hint --
     * never as an algorithm selector across families.</p>
     *
     * <p>Implementors MUST use a constant-time comparison for any
     * secret-dependent signature comparison (e.g., HMAC). See "HMAC
     * Constant-Time Comparison (Contract)" below.</p>
     *
     * @throws InvalidJWTSignatureException if the signature does not match the message
     * @throws InvalidKeyLengthException if the verifier's key is too short for the algorithm
     * @throws InvalidKeyTypeException if the key type is not compatible with the algorithm
     * @throws JWTVerifierException if an underlying JCE operation fails (e.g., NoSuchAlgorithmException)
     */
    void verify(Algorithm algorithm, byte[] message, byte[] signature);
}
```

Implementations keep the same factory method patterns:

```java
HMACSigner.newSHA256Signer(String secret)
HMACSigner.newSHA256Signer(String secret, String kid)
RSASigner.newSHA256Signer(String pemPrivateKey)
// etc.
```

### Signers / Verifiers Utility Classes

A `Signers` utility class provides dynamic algorithm-to-signer lookup for the built-in algorithms. This eliminates the switch statement users currently need when they have an `Algorithm` value at runtime (e.g., from a JWK or config file).

```java
public final class Signers {
    /**
     * Create a signer for the given algorithm using a PEM-encoded private key (for
     * asymmetric algorithms) or a secret string (for HMAC algorithms). The algorithm
     * determines interpretation: HS* algorithms treat the string as an HMAC secret;
     * all other algorithms treat it as a PEM-encoded private key.
     */
    public static Signer forAlgorithm(Algorithm algorithm, String keyMaterial) { ... }
    public static Signer forAlgorithm(Algorithm algorithm, String keyMaterial, String kid) { ... }

    /** Create a signer for an asymmetric algorithm using a PrivateKey. */
    public static Signer forAlgorithm(Algorithm algorithm, PrivateKey key) { ... }
    public static Signer forAlgorithm(Algorithm algorithm, PrivateKey key, String kid) { ... }
}

public final class Verifiers {
    /**
     * Create a verifier for the given algorithm using a PEM-encoded public key (for
     * asymmetric algorithms) or a secret string (for HMAC algorithms).
     */
    public static Verifier forAlgorithm(Algorithm algorithm, String keyMaterial) { ... }

    /** Create a verifier for an asymmetric algorithm using a PublicKey. */
    public static Verifier forAlgorithm(Algorithm algorithm, PublicKey key) { ... }

    /**
     * Returns a composite Verifier that tries each delegate in order, using the first
     * where canVerify() returns true. This replaces the varargs Verifier... parameter
     * from the 6.x JWTDecoder.decode() method.
     *
     * <p><strong>Fail-fast semantics:</strong> {@code canVerify()} answers "does this
     * verifier understand this algorithm?" -- not "is the signature valid." The first
     * verifier where {@code canVerify()} returns true gets one shot at
     * {@code verify()}. If {@code verify()} throws (e.g., invalid signature), the
     * exception propagates immediately -- subsequent verifiers are not tried. The
     * intended usage is that no two verifiers in the list should return
     * {@code canVerify() == true} for the same algorithm.</p>
     *
     * <p>If no verifier's {@code canVerify()} returns true, throws
     * {@code MissingVerifierException}.</p>
     */
    public static Verifier anyOf(Verifier... verifiers) { ... }
}
```

These cover the 14 standard algorithms only. Custom algorithm implementors construct their `Signer`/`Verifier` directly -- no registry or extension mechanism is provided at this time. The existing type-safe factory methods on each class (e.g., `HMACSigner.newSHA256Signer()`) are preserved for discoverability and compile-time safety.

**Misuse behavior.** The string-keyed factories (`forAlgorithm(Algorithm, String)`) intentionally accept different meanings for `keyMaterial` depending on the algorithm (HMAC secret vs. PEM). This is ambiguous but the least-bad option (typed wrappers were considered and rejected as more ceremony than value). Failure modes when mis-called:

- **HMAC algorithm with a PEM string:** the PEM text bytes are treated as the HMAC secret. The length check will typically *pass* (PEMs are longer than the minimum secret size), but the resulting signer/verifier is a functioning HMAC over random-looking bytes -- behavior that never matches any real counterpart. This is a correctness failure, not a safety failure: the signer produces signatures that no legitimate verifier accepts, and vice versa. Prefer the typed `HMACSigner.newSHA256Signer(...)` factory when you have an HMAC secret.
- **Asymmetric algorithm with raw HMAC bytes:** PEM parsing fails immediately with `PEMDecoderException`.

### Migration from 6.x varargs decode

```java
// 6.x: varargs verifiers
decoder.decode(token, rsaVerifier, hmacVerifier);

// 7.0: composite verifier wrapped in a VerifierResolver
decoder.decode(token, VerifierResolver.of(Verifiers.anyOf(rsaVerifier, hmacVerifier)));

// 7.0: kid-keyed lookup (preferred when both verifiers have distinct kids)
decoder.decode(token, VerifierResolver.byKid(Map.of(
    "rsa-key-1", rsaVerifier,
    "hmac-key-1", hmacVerifier
)));
```

### Key Length and Strength Enforcement

All built-in `Signer` and `Verifier` implementations enforce minimum key sizes at construction time. Tokens signed or verified with undersized keys are rejected with `InvalidKeyLengthException` before any signature operation is attempted.

| Algorithm | Minimum key size | Source |
|-----------|------------------|--------|
| `HS256` | 256-bit (32-byte) HMAC secret | RFC 7518 §3.2 |
| `HS384` | 384-bit (48-byte) HMAC secret | RFC 7518 §3.2 |
| `HS512` | 512-bit (64-byte) HMAC secret | RFC 7518 §3.2 |
| `RS256`, `RS384`, `RS512` | 2048-bit RSA modulus | RFC 8725 §3.5; library accepts 2047-bit to handle edge-case key generation |
| `PS256`, `PS384`, `PS512` | 2048-bit RSA modulus | RFC 8725 §3.5 |
| `ES256` | P-256 curve (256-bit) | RFC 7518 §3.4 |
| `ES384` | P-384 curve (384-bit) | RFC 7518 §3.4 |
| `ES512` | P-521 curve (521-bit) | RFC 7518 §3.4 |
| `ES256K` | secp256k1 curve (256-bit) | RFC 8812 |
| `Ed25519` | Ed25519 curve | RFC 8037 / RFC 9864 |
| `Ed448` | Ed448 curve | RFC 8037 / RFC 9864 |

EC and EdDSA signers/verifiers additionally validate that the key's curve matches the algorithm (e.g., `ES256` with a P-384 key is rejected as `InvalidKeyTypeException`).

This enforcement cannot detect low-entropy keys. A 32-byte HMAC secret of all zeros passes the length check; callers are responsible for generating keys with sufficient entropy (use `JWTUtils.generateSHA256_HMACSecret()` or equivalent).

### HMAC Constant-Time Comparison (Contract)

Any `Verifier` implementation performing HMAC (or any secret-dependent) signature comparison MUST use a constant-time comparison function. The built-in `HMACVerifier` uses `java.security.MessageDigest.isEqual()`, which is specified constant-time on modern JDKs.

Using `Arrays.equals()`, `byte[].equals()`, or any short-circuiting comparison is a timing-attack vulnerability and is prohibited by this contract. This requirement is documented here rather than enforced because `Verifier` is a public interface and implementors must take responsibility for this property.

### Signer / Verifier / `kid` Binding

A `Signer` or `Verifier` instance represents a tight binding between a specific **key**, a specific **algorithm**, and an optional **`kid`**. Callers MUST treat these as coupled:

- Never reuse a `Verifier` instance across different key material.
- Never construct a `Signer` whose `algorithm()` and `kid()` are associated with different underlying keys.
- When indexing verifiers by `kid` (the `Map<String, Verifier>` decode path), each `kid` in the map must correspond to a verifier whose `canVerify()` returns true for exactly one algorithm family -- never wire a `kid` to a verifier that accepts multiple algorithm families (e.g., a custom `anyOf`-style verifier).

Mixing these bindings is a common source of algorithm-confusion vulnerabilities and produces undefined behavior. RFC 8725 §3.1 covers this class of issue; the library's defense (algorithm check via `canVerify`) catches most cases, but a misconfigured `Verifier` implementation can defeat it. Implementor discipline is required.

## 7. KeyType (Interface)

Same pattern as Algorithm -- replaces the current `KeyType` enum with an interface for extensibility.

```java
public interface KeyType {
    /** JWK "kty" parameter value. */
    String name();

    KeyType RSA = new StandardKeyType("RSA");
    KeyType EC  = new StandardKeyType("EC");
    KeyType OKP = new StandardKeyType("OKP");
    KeyType OCT = new StandardKeyType("oct");

    /** Look up by name. Returns a standard constant if matched, otherwise a new instance. */
    static KeyType of(String name) { ... }
}
```

`StandardKeyType` is a package-private class with `equals`/`hashCode` based on `name()`, same pattern as `StandardAlgorithm`. `KeyType.of()` interns standard constants the same way `Algorithm.of()` does.

`KeyType.OCT` represents symmetric keys per RFC 7517 Section 6.4. This ensures the JWK parser does not choke on `"kty": "oct"` entries in a JWKS response. Full symmetric JWK support (building/parsing `oct` keys) may be added in a future release, but the `KeyType` constant is defined now for forward compatibility.

**RSA-PSS keys:** No separate `KeyType.RSA_PSS` is defined. The IANA "JSON Web Key Types" registry does not include `"RSASSA-PSS"` as a `kty` value -- RSA-PSS keys use `"kty": "RSA"` on the wire, the same as PKCS#1 v1.5 keys. The algorithm distinction is in the `"alg"` parameter (e.g., `"PS256"`), not in `kty`. Internally, Java's `KeyFactory` uses `"RSASSA-PSS"` as an algorithm identifier for PSS-specific keys -- the `JSONWebKeyBuilder` and `JSONWebKeyParser` handle this distinction without exposing it through `KeyType`.

### Private Key Material Handling

`JSONWebKey` can hold private key material (`d`, `dp`, `dq`, `p`, `q`, `qi` for RSA; `d` for EC/OKP). Two defensive measures:

**1. Redacted `toString()`.** `toString()` replaces all private-material fields with `"***"`, so that a careless `log.debug("key={}", jwk)` or a Java debugger variables panel does not accidentally surface private key bytes. This is not security-critical (callers with JWK access can inspect fields directly), but it closes the most common foot-gun.

The current code uses `toString()` extensively in test failure messages and logs -- migrating it to always-redact prevents regressions where new debug logging inadvertently leaks private material.

**2. `toPublicJSONWebKey()`.** Returns a copy with all private fields nulled. Intended for serving a JWKS endpoint from a private key without manually filtering fields:

```java
// Build keypair, serve the public half on /.well-known/jwks.json
JSONWebKey privateKey = JSONWebKey.build(privateKeyPem);
JSONWebKey publicKey = privateKey.toPublicJSONWebKey();
// publicKey.d() is null, publicKey.n() and publicKey.e() are preserved
```

Note: `x5c` (certificate chain) is not treated as private material -- certificates are inherently public. `key_ops` is not filtered; callers who want to remove sign-specific ops (`sign`, `decrypt`, `unwrapKey`, `deriveKey`, `deriveBits`) from the public copy can do so manually via the builder.

## 9. PEM/DER Enhancements: X.509 Without Private Classes

### Goal

Parse and generate X.509 certificates using our own DER encoder/decoder, eliminating the dependency on `java.security.cert.CertificateFactory` where possible. This gives us full control over the certificate structure and removes reliance on JDK internals.

### Current State

- The DER infrastructure (`DerInputStream`, `DerOutputStream`, `DerValue`, `Tag`, `ObjectIdentifier`) is mature and handles key encoding/decoding well.
- `PEMDecoder` handles X.509 certificate PEM blocks but delegates to `CertificateFactory.getInstance("X.509")` for the actual parsing.
- `PEMEncoder` can encode certificates in PEM format.
- `ObjectIdentifier` already defines OIDs for common algorithms (RSA, EC, EdDSA).

### Prior Art: moreDerEncoding branch

The `moreDerEncoding` branch on `fusionauth/fusionauth-jwt` contains incomplete but directionally correct work on this. Key pieces to carry forward:

**DerValue factory methods** (partially implemented, need porting):

DerValue.newGeneralizedTime(Date date)     // "yyyyMMddHHmmss'Z'" format
DerValue.newUTCTime(Date date)             // "yyMMddHHmmss'Z'" format
DerValue.newNull()                         // Tag.Null with empty body
DerValue.newASCIIString(String s)          // Tag.PrintableString
DerValue.newUTF8String(String s)           // Tag.UTFString (0x0C)
DerValue.getBitStringBytes()               // strips padding byte, returns raw bytes
```

**Tag constants** (need adding):

// Set and Sequence already exist but need constructed-form bit set:
Tag.Set = 17 | 0b00100000      // always constructed

```

**ObjectIdentifier.encode(String)** -- converts dot-notation OID string to DER byte array. The branch implementation handles the first two components (40*a + b encoding) and multi-byte encoding for components >= 128. Needs review for components >= 16384 (three-byte encoding).

**X.509 Certificate Generation** (from `KeyUtils.generateX509CertificateFromKey`):

The branch builds TBSCertificate as a DER SEQUENCE following RFC 5280:

```
Certificate ::= SEQUENCE {
    tbsCertificate       TBSCertificate,
    signatureAlgorithm   AlgorithmIdentifier,
    signatureValue       BIT STRING
}

TBSCertificate ::= SEQUENCE {
    version         [0] EXPLICIT Version DEFAULT v1,     -- context tag 0xA0
    serialNumber        CertificateSerialNumber,         -- INTEGER
    signature           AlgorithmIdentifier,             -- SEQUENCE { OID, NULL }
    issuer              Name,                            -- SEQUENCE { SET { SEQUENCE { OID, PrintableString } } }
    validity            Validity,                        -- SEQUENCE { UTCTime, UTCTime }

    subjectPublicKeyInfo SubjectPublicKeyInfo            -- raw bytes from PublicKey.getEncoded()

```

The branch constructs this with nested `DerOutputStream` calls. The approach is correct but incomplete:
- Signing works (uses `java.security.Signature`) but the algorithm is hardcoded
- Time encoding uses UTCTime for dates before 2050, GeneralizedTime after (per RFC 5280)
- The result is passed back through `CertificateFactory` to get an `X509Certificate` object -- this is acceptable since CertificateFactory is standard JDK, not private API. The goal is to avoid private classes for *construction*, not necessarily for the return type.
- Missing: EC-specific algorithm identifier sequences (EC uses the curve OID alongside the algorithm OID, not NULL)
- Missing: EdDSA support
- Missing: Extensions (v3 certificates)

**X.509 Certificate Parsing (DER -> structured fields):**

- Issuer and subject distinguished names
- Validity period (notBefore, notAfter)

Primary use case: extracting the public key and metadata from PEM-encoded certificates in environments where `CertificateFactory` may not be desirable or for round-trip fidelity.

### Scope

The goal is to handle the common X.509 use cases in JWT/JWK workflows -- generating self-signed certificates, extracting public keys from certificates, and `x5c` chain handling. Full PKIX path validation, CRL/OCSP checking, and v3 extensions are out of scope for 7.0 but the DER infrastructure should be extensible enough to add them later.

### JWK Thumbprint Canonicalization (RFC 7638)

RFC 7638 §3 requires a **specific canonical JSON form** for thumbprint computation:

1. Only the REQUIRED members for the key type are included.
2. Members are ordered lexicographically by Unicode code point of the JSON member name.
3. No whitespace is permitted.
4. UTF-8 encoding.

This form is **incompatible with a pluggable `JSONProcessor`**: a user-supplied implementation (Jackson, Gson, a custom encoder) might reorder keys, insert whitespace, or produce a different string representation of the same logical JSON value. If two services computing the same thumbprint disagree on canonicalization, `kid` lookups break silently across the infrastructure.

The 6.x implementation relies on Jackson's default behavior + `LinkedHashMap` insertion order to produce canonical JSON. This works only by accident: it requires (a) the serializer iterates the map in insertion order, (b) the serializer adds no whitespace, (c) nobody has reconfigured the `ObjectMapper` with `SerializationFeature.ORDER_MAP_ENTRIES_BY_KEYS`, (d) the `LinkedHashMap` contract is preserved. None of these are guaranteed by the `JSONProcessor` contract in 7.0.

**7.0 implementation:** `JWTUtils.generateJWS_kid(...)` uses an internal `CanonicalJSONWriter` (not `JSONProcessor`) that produces RFC 7638-compliant output unconditionally. The writer is ~40 LOC, supports only the primitive types needed for JWK thumbprint members (strings, numbers, booleans, null), and lives under `org.lattejava.jwt.internal`. It is package-private and never exposed via public API.

Required members per key type (per RFC 7638 §3.2 and RFC 8037 §2):

| `kty` | Required members (in lex order) |
|-------|---------------------------------|
| `RSA` | `e`, `kty`, `n` |
| `EC` | `crv`, `kty`, `x`, `y` |
| `OKP` (Ed25519/Ed448) | `crv`, `kty`, `x` |
| `oct` | `k`, `kty` |

**Input canonicalization is the caller's responsibility.** `CanonicalJSONWriter` canonicalizes key *ordering* and whitespace; it emits string *values* exactly as stored on the `JSONWebKey`. RFC 7518 §6.3.1.1 specifies a canonical byte encoding for RSA `n`/`e` (unsigned big-endian, no leading zero octets); similar rules apply to EC `x`/`y` and OKP `x`. If a `JSONWebKey` is built via `JSONWebKey.build(PublicKey)`, the library produces RFC 7518-canonical bytes automatically. If a `JSONWebKey` is built via the `Builder` with hand-constructed base64url strings, leading-zero padding or non-minimal encodings produce a *different* thumbprint than the canonical form -- callers who interoperate with external services MUST supply canonically-encoded values. This matches the behavior of jjwt, nimbus-jose-jwt, and auth0 java-jwt.

### Internal SHAKE256 Implementation (for OIDC Ed448 `at_hash` / `c_hash`)

**Motivation.** OpenID Connect `at_hash` and `c_hash` are computed with an algorithm-matched digest. For Ed448 the matched digest is SHAKE256 (FIPS 202). Stock `SunEC` uses SHAKE256 internally to implement Ed448 sign/verify but does **not** register it as a public `java.security.MessageDigest` service on any shipping JDK. `MessageDigest.getInstance("SHAKE256")` therefore throws `NoSuchAlgorithmException` on a stock JDK even when Ed448 sign/verify itself works. To keep the library zero-dependency for the Ed448 OIDC-hashing path, the library ships a minimal FIPS 202 SHAKE256 implementation and prefers a JCE-registered provider when one is available.

**Design.**

- New package-private class: `org.lattejava.jwt.internal.SHAKE256`.
- Self-contained implementation of FIPS 202 Keccak-f[1600] plus the SHAKE256 sponge (rate = 1088 bits, capacity = 512 bits, domain separator `0x1F`).
- Not registered as a JCE `Provider` or `MessageDigest` service. Called directly from `OpenIDConnect.generate_hash` in the `Ed448` branch; the SHA-2 branches continue to route through `MessageDigest.getInstance`.
- API is intentionally minimal:
  ```java
  final class SHAKE256 {
      // Returns `outputBytes` of SHAKE256 output for the given input.
      // Prefers a JCE-registered "SHAKE256" provider if one is available and
      // passes an internal self-test; otherwise falls back to the bundled
      // FIPS 202 implementation.
      static byte[] digest(byte[] input, int outputBytes);
  }
  ```
- The Ed448 branch of `OpenIDConnect.generate_hash` requests 57 bytes (228 bits), matching `114 * 8 / 2 = 456 / 8 = 57`. Today's code slices the digest with `Arrays.copyOfRange`; the new implementation directly squeezes the requested length.

**Provider preference (FIPS-aware).** Deployments that need FIPS 140 compliance will register a validated provider such as **BC-FIPS** (BouncyCastle FIPS, 1.0.2+) that exposes SHAKE256 via the standard JCA `MessageDigest` service. Calling our bundled implementation directly in that environment would bypass the FIPS boundary for `at_hash`/`c_hash`. To preserve FIPS compliance when a validated provider is present, `SHAKE256.digest(...)` **prefers the JCE-registered provider** and only falls back to the internal implementation when none is available.

- **Lazy probe.** On the first call to `SHAKE256.digest(...)`, the class probes `MessageDigest.getInstance("SHAKE256")`. Class-load-time probing is not used because BC / BC-FIPS is typically registered during application startup but after class-load of unrelated library classes; lazy probing ensures we see providers registered at any point before the first OIDC hash is computed.
- **Self-test at probe time.** Probing is not limited to `getInstance` success. We execute a one-shot `digest(buf, 0, 57)` against a known SHAKE256 KAT pair (input + expected 57-byte output) at probe time and compare the result with `MessageDigest.isEqual`. If the provider registers SHAKE256 but its variable-length digest is broken, incomplete, or inconsistent, the self-test fails and we fall back to the internal implementation. This protects against partial/incorrect provider registrations.
- **Caching.** On successful probe we cache the `Provider` (not the `MessageDigest` instance — `MessageDigest` is not thread-safe and cannot be reused across threads). Subsequent calls execute `MessageDigest.getInstance("SHAKE256", cachedProvider)` to bind to the same provider without rescanning. The cache is a `volatile` reference; a benign race on first call results in two threads probing independently and arriving at the same answer.
- **Lifetime.** The decision (JCE provider vs. internal) is cached for the lifetime of the VM. Providers are rarely unregistered in practice; if a caller removes a provider mid-run, the library continues to use the cached reference until VM exit.
- **No registration.** The library never calls `Security.addProvider` or `Security.insertProviderAt`. It does not register its internal SHAKE256 as a JCA service. The only JCA interaction is a read-only `MessageDigest.getInstance` probe.

**Drift / conformance risk.** SHAKE256 is a deterministic, fully specified function (FIPS 202 §6.2). Any conformant implementation produces bit-identical output for identical input — there is no implementation latitude (no blinding, no keying, no parameter choices). Both the bundled implementation and any JCA-registered provider (BC, BC-FIPS, etc.) produce identical results for identical inputs, so the choice between them is a *trust boundary* decision (FIPS compliance), not a behavioral one. Conformance of the bundled implementation is verified with the NIST CAVP SHAKE256 Known-Answer Test vectors in the test plan; cross-validation against BouncyCastle over randomized inputs provides additional assurance.

**FIPS compliance note.** Preferring the JCE provider makes *FIPS compliance possible* when the caller deploys a FIPS-validated provider and enables FIPS mode — it does not make the library itself FIPS-validated (libraries are not validated; cryptographic modules are). Callers remain responsible for provisioning BC-FIPS (or equivalent), configuring `java.security` for FIPS mode, and validating their full application stack.

**Provenance.** The bundled implementation is derived from a public-domain reference (e.g., XKCP or `tiny_sha3`) adapted to Java; the source is credited in the file header. No third-party runtime dependency is introduced.

**Non-goal.** The library does not expose `SHAKE256` as a public API, does not register it as a JCA service, and does not support SHAKE128, cSHAKE, or other Keccak variants. The implementation serves exactly one internal caller.

## 13. Performance Considerations

- **Map intermediary overhead:** Building a `LinkedHashMap` with ~7-10 entries is cheap (hundreds of nanoseconds). The trade-off vs. Jackson's reflection-based annotation processing is likely neutral or favorable for small payloads.
- **Built-in parser:** No reflection, no annotation processing, no module system. For JWT-sized payloads (~200-500 bytes of JSON), a hand-rolled parser can match or beat Jackson.
- **BigInteger/BigDecimal overhead:** Slightly more allocation than Long/Double, but negligible for JWT-sized payloads. The consistency benefit outweighs the cost.
- **Crypto dominates:** Signature operations (RSA, EC, HMAC) are orders of magnitude slower than JSON serialization. The JSON layer is not the bottleneck.
- **Benchmark plan:** Once implemented, benchmark against the current Jackson-based version and against other JWT libraries (JJWT, auth0, nimbus) for encode/decode throughput.

## 15. RFC 8725 Compliance

[RFC 8725](https://www.rfc-editor.org/rfc/rfc8725) — "JSON Web Token Best Current Practices" (BCP 225) — is the IETF's consolidated guidance on avoiding the known JWT pitfalls. This library aligns with RFC 8725 as follows. Items marked "Caller responsibility" are outside the library's purview but mentioned for completeness.

| RFC 8725 § | Topic | Library behavior |
|-----------|-------|------------------|
| §2.1 | Weak signatures and insufficient signature validation | Covers the "none" alg attack, algorithm confusion, and signature stripping. No built-in verifier handles `alg: none` -- the decoder throws `MissingVerifierException` unless the caller explicitly wires a verifier for it (see [Section 1](#1-algorithm-interface) "Security: the 'none' algorithm"). All built-in `Verifier` implementations have `canVerify()` that returns true only for their own algorithm family -- RSA/EC/EdDSA verifiers reject HS* header values, and the HMAC verifier's `verify()` fails on forged signatures. A token missing the signature segment throws `MissingSignatureException`. |
| §2.2 | Weak symmetric keys | Minimum HMAC secret sizes enforced at construction time per RFC 7518 §3.2 -- see [Section 6](#6-signer--verifier) "Key Length and Strength Enforcement". |
| §2.6 | Multiplicity of JSON encodings | Base64URL strict decoding rejects lenient encodings (see [Section 5](#5-encoder--decoder) "Base64URL Strictness"). `LatteJSONProcessor` rejects duplicate JSON keys by default. For RFC 7638 JWK thumbprints the library uses a separate internal `CanonicalJSONWriter` unaffected by user `JSONProcessor` choice (see [Section 10](#10-jwtutils)). |
| §2.8 | Cross-JWT confusion | Addressed via `expectedType` (§3.11) and `expectedAlgorithms` (§3.1). Callers enforce explicit type and algorithm per usage site. |
| §3.1 | Perform algorithm verification | Decoder checks `verifier.canVerify(header.alg)` before calling `verify()`. Optional `expectedAlgorithms` set provides defense-in-depth whitelist that runs before verifier resolution. |
| §3.2 | Use appropriate algorithms | All standard algorithms in [Section 1](#1-algorithm-interface) are from the IANA JOSE registry. Deprecated `"EdDSA"` identifier is not a standard constant per RFC 9864. No `Algorithm.NONE` constant. |
| §3.3 | Validate all cryptographic operations | `Verifier.verify()` throws on failure and the library does not silently fall through. Signature verification runs before payload parsing, so an unauthenticated payload is never surfaced through a `JWT` object. |
| §3.4 | Validate cryptographic inputs | Base64URL strictness rejects malformed segments. `LatteJSONProcessor` rejects duplicate keys and enforces depth limits. Token size is capped via `maxTokenBytes`. `crit` structural validation rejects malformed critical-parameter lists (see [Section 3](#3-header-immutable--builder)). |
| §3.5 | Ensure cryptographic keys have sufficient entropy | Library enforces minimum key sizes (see [Section 6](#6-signer--verifier) "Key Length and Strength Enforcement"). **Caller responsibility:** key entropy itself cannot be measured; use `JWTUtils.generate*()` helpers which use `SecureRandom`. |
| §3.6 | Avoid compression of encryption inputs | N/A -- JWE is out of scope for 7.0. |
| §3.7 | Use UTF-8 | All JSON and base64url decoding uses UTF-8 explicitly via `StandardCharsets.UTF_8`. |
| §3.8 | Validate issuer and subject | **Caller responsibility** -- use the post-decode `Consumer<JWT> validator` parameter to check `iss`/`sub` per application policy. |
| §3.9 | Use and validate audience | **Caller responsibility** -- `jwt.audience()` returns `List<String>`; check membership per application policy. Library always exposes `aud` as a list to eliminate the "string vs. array" branch at the call site (see [Section 2](#2-jwt-immutable--builder) "Audience Handling"). |
| §3.10 | Do not trust received claims | Decoder validates signature before claims are parsed into the `JWT` object. The post-decode `Consumer<JWT> validator` hook runs after signature/time validation, with explicit guidance that application-level claim validation is the caller's job. |
| §3.11 | Use explicit typing | Encoder defaults `typ` to `"JWT"`. Decoder supports `expectedType()` for applications that want to pin `typ` to a specific value (e.g., `"at+jwt"` per RFC 9068, `"dpop+jwt"` per RFC 9449). |
| §3.12 | Use mutually exclusive validation rules for different kinds of JWTs | Supported via `expectedType` + `expectedAlgorithms` combination. Different decoder instances can be configured with mutually-exclusive rule sets for access tokens vs. ID tokens vs. DPoP proofs. |

Note: `crit` unknown-parameter handling is defined by RFC 7515 §4.1.11 (not RFC 8725). See [Section 3](#3-header-immutable--builder) "Critical Header Parameter" for that contract.

### What this library does NOT do

The library deliberately stops at mechanism, not policy. The following are documented non-features — callers are expected to handle them in application code via the `Consumer<JWT> validator` hook:

- Issuer (`iss`) validation
- Audience (`aud`) validation against a known service identifier
- Subject (`sub`) format validation
- Maximum token lifetime policy
- Replay-protection / `jti` tracking
- Nonce (`nonce`) validation for OIDC

These are intentionally not built in because they are application-specific. Providing them as library defaults would mean either (a) getting them wrong for most users, or (b) exposing a wide configuration surface that duplicates what a 5-line `Consumer<JWT>` does cleanly.
